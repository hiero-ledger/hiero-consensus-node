# Task schedulers and queues

## Prerequisites

- [`pass1-04-event-creation-under-stress`](./pass1-04-event-creation-under-stress.md) — the wiring framework's bounded schedulers are the substrate where stress first manifests as queue depth; the health monitor reads that depth via `TaskScheduler.getUnprocessedTaskCount()` and `TaskScheduler.getCapacity()`, and downstream reaction sites act on the resulting signal. The mental model from that scenario is: every consensus-layer component sits behind a scheduler whose queue depth is observable, and the queue is what fills first when something falls behind.

If you feel solid on that — that bounded queues per scheduler are the raw signal the rest of the stress story runs on — you have the entry condition for this lesson, which goes one level down to the scheduler's own machinery.

## Incoming retrieval probes

This section is an authorial signal to the tutor, not a script of opening quiz prompts. The tutor does not run these as session-open quizzes; their purpose is to flag what to listen for during the entry self-assessment and to consolidate against if the learner surfaces these concepts during delivery.

- **Bounded queue per scheduler** (from `pass1-04-event-creation-under-stress`). Canonical statement to consolidate against: every component in the consensus layer sits behind a scheduler that owns a queue with a declared capacity, and the queue's unprocessed-task count is the raw observation from which the health monitor produces the unhealthy-duration signal.
- **Stress shows up as queue depth, not as a thrown exception** (from `pass1-04-event-creation-under-stress`, Stop 1). Canonical statement: under the default `platform.wiring.hardBackpressureEnabled = false`, an over-capacity scheduler does not refuse work — the queue is allowed to grow, and the health monitor turns the over-capacity state into a duration the rest of the system reacts to. This sets up the contrast in [Queue mechanics](#queue-mechanics) below between the *insertion* behaviour (which the wire-level mechanism controls) and the *signal* behaviour (which the health monitor controls).

If the learner does not surface these in self-assessment and the tutor sees no signs of either during the early walk of the Mechanism section, that is fine — the spine reintroduces both at the point where they are load-bearing for the new material.

## Misconception watchlist

- **"The component owns its queue."** Imported from servlet/handler frameworks or from naive actor descriptions. In this codebase the queue is a property of the *scheduler*, and multiple input wires on the same scheduler share a single queue and a single on-ramp counter. Listen for utterances like "the deduplicator's queue" or "the orphan buffer's queue" used as if each handler owned its own backlog. Correction in line: the scheduler is the backlog-holder; "the deduplicator's queue" is shorthand for "the queue of the scheduler running the deduplicator component," and if a second input wire were attached to that scheduler it would share the same backlog. Anchored at [`TaskScheduler.getUnprocessedTaskCount()`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/TaskScheduler.java#L186-L201) — one counter per scheduler, regardless of how many input wires are bound.
- **"`SEQUENTIAL` means a dedicated thread."** Imported from textbook actor models or single-thread executors. Listen for "the SEQUENTIAL deduplicator's thread" or "each SEQUENTIAL scheduler has its own thread." Correction: `SEQUENTIAL` schedulers run on a shared fork-join pool one task at a time, with a happens-before edge between consecutive tasks; the OS thread changes between tasks. `SEQUENTIAL_THREAD` is the variant that pins a dedicated thread. Anchored at [`TaskSchedulerType.SEQUENTIAL`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/builders/TaskSchedulerType.java#L8-L19) javadoc.
- **"`put` blocks at capacity, full stop."** Imported from blocking-queue intuitions. Listen for "the producer always blocks" without naming the soldering. Correction: `put` blocks *only if* the input wire is on a scheduler with bounded capacity *and* the producer reached the scheduler via a `PUT` solder (the default). `INJECT` solders bypass capacity entirely, and `OFFER` solders drop instead of block. The wire-level mechanism is the choice between these three, edge by edge — covered in detail in `c0-02-soldering-and-wires` but acknowledged here so the learner does not over-fix on `put`'s blocking semantics in isolation. Anchored at [`InputWire.put` / `offer` / `inject`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/wires/input/InputWire.java#L66-L95) and [`SolderType`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/wires/SolderType.java).
- **"Default capacity is unbounded."** Imported from `ExecutorService` defaults or general-purpose queue defaults. Listen for "unless we configure it, the queue grows without bound" used as a positive statement about the framework. Correction: the `AbstractTaskSchedulerBuilder` default for `unhandledTaskCapacity` is `1`, not unbounded. Consensus-layer schedulers always override this via `TaskSchedulerConfiguration` — but the framework default itself is a small positive number, not a marker for unbounded. The `TaskSchedulerConfiguration` record's javadoc uses `0` as the "unbounded" marker, but that is the configuration-level encoding, not the builder default. Anchored at [`AbstractTaskSchedulerBuilder.unhandledTaskCapacity`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/builders/internal/AbstractTaskSchedulerBuilder.java#L44).

### Pre-training

The pre-training list is exposition — the tutor states each term and moves on; it does not pose a definition quiz on this list.

- **`TaskScheduler<OUT>`** — the framework primitive that owns one queue, one thread-execution policy, and one primary `OutputWire<OUT>`. The class is abstract; the runtime instance is one of `SequentialTaskScheduler`, `ConcurrentTaskScheduler`, `SequentialThreadTaskScheduler`, or a `DIRECT`/`DIRECT_THREADSAFE`/`NO_OP` variant — chosen by `TaskSchedulerType`. See [`TaskScheduler`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/TaskScheduler.java).
- **`TaskSchedulerType`** — an enum with six values that selects the threading policy. Defined and documented at [`TaskSchedulerType`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/builders/TaskSchedulerType.java).
- **`TaskSchedulerConfiguration`** — a record carrying the configuration knobs for one scheduler: type, capacity, four metric toggles, flushability, and squelchability. Modules read these from a typed `*WiringConfig` and pass them to `ComponentWiring`'s constructor. See [`TaskSchedulerConfiguration`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/builders/TaskSchedulerConfiguration.java).
- **On-ramp counter** — the counter inside the scheduler that increments on every accepted enqueue and decrements after the handler returns. It is what `getUnprocessedTaskCount()` exposes; it is what `flush()` waits on; it is what the capacity is checked against on `put`. Anchored at `SequentialTaskScheduler.put/offer/inject` (`onRamp.onRamp()` / `attemptOnRamp()` / `forceOnRamp()`) and [`SequentialTaskScheduler.flush`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/internal/SequentialTaskScheduler.java#L152-L157).
- **Primary `OutputWire<OUT>`** — every scheduler owns exactly one. Built in `TaskScheduler`'s constructor at `primaryOutputWire = buildPrimaryOutputWire(model, name)` and returned by [`TaskScheduler.getOutputWire()`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/TaskScheduler.java#L116-L126). The primary wire receives whatever the bound handler returns and participates in the on-ramp accounting; secondary output wires exist (for fan-out side-channels) but are not in scope for this lesson.
- **`InputWire<IN>`** — the entry point of a component. Carries the three put-paths `put`, `offer`, and `inject`. See [`InputWire`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/wires/input/InputWire.java). One scheduler may carry several input wires; all share the same on-ramp counter and the same capacity.
- **`ComponentWiring<COMPONENT, OUT>`** — the canonical wrapper that consensus-layer modules use to build a scheduler from a `TaskSchedulerConfiguration` and bind a component to it. The framework primitive `WiringModel.schedulerBuilder(name).withType(...).build()` is rarely used directly — it lives inside `ComponentWiring`'s constructor. See [`ComponentWiring`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/component/ComponentWiring.java).

Also useful as referents for the misconception watchlist, but not directly walked through in the spine: `SolderType` (the enum that pairs `PUT`/`OFFER`/`INJECT` with the three `InputWire` paths; covered in detail in `c0-02-soldering-and-wires`); the on-ramp counter's `attemptOnRamp` / `forceOnRamp` behaviour (covered structurally below at [Queue mechanics](#queue-mechanics)).

### Chunk 1 — What a `TaskScheduler` owns {#chunk-scheduler-spine}

`moment_id: scheduler-spine` (engagement-move inventory below)

A `TaskScheduler<OUT>` owns three things and no others: a queue (on-ramp counter plus the thread-execution machinery that drains it), a thread-execution policy (named by its `TaskSchedulerType`), and a primary `OutputWire<OUT>` that downstream code reads. Every other piece of the wiring framework — input wires, transformers, splitters, soldering — is either attached to a scheduler or hanging off its primary output wire.

The three are visible at construction in [`TaskScheduler`'s constructor](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/TaskScheduler.java#L65-L88): the `type` field captures the policy, `primaryOutputWire = buildPrimaryOutputWire(model, name)` builds the output wire, and the queue's on-ramp/off-ramp counters are passed in by the concrete subclass — see [`SequentialTaskScheduler`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/internal/SequentialTaskScheduler.java#L52-L82) which receives `onRamp` and `offRamp` as constructor parameters and stores `capacity` as its only own field beyond what the parent holds.

> **Load-bearing line.** *The queue, the threading policy, and the primary output wire are properties of the scheduler — not of the component, not of any individual input wire.* This is the load-bearing structural fact for the rest of the lesson, the cluster, and most of how stress propagates through the consensus layer. It is the fact behind why `getUnprocessedTaskCount()` is one number per scheduler (anchored at [`TaskScheduler.getUnprocessedTaskCount`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/TaskScheduler.java#L186-L201)), why capacity is one number per scheduler (anchored at [`TaskScheduler.getCapacity`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/TaskScheduler.java#L214-L220)), and why the health monitor watches schedulers rather than wires.

A component can have several input wires, and the framework's idiomatic use does — `EventDeduplicator` has `handleEvent`, `setEventWindow`, and `clear` input wires soldered into its scheduler. All three input wires put data into the same on-ramp counter and the same queue. The deduplicator's "queue" is one queue, shared across its three input wires, holding the union of arrivals on all of them, with the scheduler's policy deciding which one runs next.

There is a structural reason for this. The scheduler's policy — `SEQUENTIAL`, `CONCURRENT`, etc. — is a property of *how tasks execute*, and any meaningful ordering or concurrency contract has to apply to the union of arrivals from all input wires, not to each input wire independently. If the deduplicator's `handleEvent` and `clear` could each have their own queue with no shared discipline, a clear command could land between two events and execute on a different thread from them, violating the happens-before edge `SEQUENTIAL` is supposed to give. One queue per scheduler is what makes the policy a coherent claim about the component as a whole.

### Chunk 2 — The six `TaskSchedulerType` values {#chunk-type-selection}

`moment_id: type-selection` (engagement-move inventory below)

`TaskSchedulerType` (defined at [`TaskSchedulerType`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/builders/TaskSchedulerType.java)) has six values. Two run sequentially on a shared pool or a dedicated thread; one runs in parallel; two skip the queue and run on the caller's thread; one discards everything. The list below paraphrases the enum's javadoc; defer to the source for the authoritative phrasing.

- **`SEQUENTIAL`** — fork-join pool, one task at a time, happens-before edge between consecutive tasks (javadoc at [`TaskSchedulerType.SEQUENTIAL`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/builders/TaskSchedulerType.java#L8-L19)). The OS thread changes between tasks, but the happens-before guarantee means a write performed by one task is visible to the next. This is the default if `TaskSchedulerConfiguration.type` is unset and the default field value in `AbstractTaskSchedulerBuilder` ([`type = TaskSchedulerType.SEQUENTIAL`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/builders/internal/AbstractTaskSchedulerBuilder.java#L42)).
- **`SEQUENTIAL_THREAD`** — dedicated thread, one task at a time. Same ordering and happens-before guarantee as `SEQUENTIAL`; differs in that the thread is bound to this scheduler and is not shared with the fork-join pool.
- **`CONCURRENT`** — fork-join pool, tasks may execute in parallel, no ordering guarantee.
- **`DIRECT`** — no queue at all; the task runs synchronously on the caller's thread. Subject to graph-walk validation: only a single sequential producer is permitted as the upstream source of a `DIRECT` scheduler, and a `CONCURRENT` upstream is forbidden. The constraints are enforced at graph-start time by the framework's validation pass — see the enum's javadoc at [`TaskSchedulerType.DIRECT`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/builders/TaskSchedulerType.java#L25-L51) for the algorithm; the validation runs from `StandardWiringModel.start()` (anchored at [`StandardWiringModel.start`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/model/StandardWiringModel.java#L205-L231), which calls `checkForCyclicalBackpressure`, `checkForIllegalDirectSchedulerUsage`, `checkForUnboundInputWires`).
- **`DIRECT_THREADSAFE`** — same as `DIRECT` but the handler is required to be threadsafe; without that constraint the validation is relaxed (the handler may be invoked from multiple producers). The framework cannot check threadsafety; the contract is on the author. This is the type used internally by `WireTransformer` and `WireListSplitter` (anchored at [`WireTransformer` line 44](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/transformers/WireTransformer.java#L43-L45) and [`WireListSplitter` line 37](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/transformers/WireListSplitter.java#L36-L38)).
- **`NO_OP`** — discards everything. Wires into and out of a `NO_OP` scheduler are effectively non-existent at runtime; useful for toggling a component off without removing its wiring. Anchored at the enum's javadoc at [`TaskSchedulerType.NO_OP`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/builders/TaskSchedulerType.java#L62-L68), with the consumer-side short-circuit visible in `BindableInputWire`'s `noOp` branch (anchored at [`BindableInputWire` lines 55–60 and the early returns in `bind`/`bindConsumer`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/wires/input/BindableInputWire.java#L55-L80)).

The choice of type is not a default; the topic file is explicit that "there is no canonical scheduler type for new components — the choice is made case-by-case and lives in the component's `*WiringConfig` record" (anchored at [`wiring-framework.md` section "Core abstractions → TaskScheduler / TaskSchedulerType"](../../architecture/topics/wiring-framework.md#taskscheduler--taskschedulertype)). The choice falls out of two structural questions about the component: does its state require single-threaded access (then `SEQUENTIAL` or `SEQUENTIAL_THREAD`), or is the handler stateless or independently-threadsafe (then `CONCURRENT`, or `DIRECT_THREADSAFE` when the task is small enough that even queueing overhead is too much)? `DIRECT` is reserved for the special case where a single sequential upstream is the natural producer, and `NO_OP` for the toggle-off case.

For an example sized against this lesson: the `OrphanBuffer` runs on a `SEQUENTIAL` scheduler (per the topic file in [event-intake.md, "5. Orphan buffer"](../../architecture/topics/event-intake.md#5-orphan-buffer-topological-ordering)) because the orphan buffer must release events in topological order — and the release walk has to run on a single thread for that ordering to hold, so concurrent arrivals are not an option. The hasher and the signature validator run `CONCURRENT` because hashing and signature verification are independent per event and benefit from parallelism. The selection is anchored on a structural property of the handler (does it have an ordering requirement or not), not on a default.

### Chunk 3 — Queue mechanics: capacity, on-ramp, put / offer / inject {#queue-mechanics}

`moment_id: queue-mechanics` (engagement-move inventory below)

The scheduler's queue is a counter, not a literal `BlockingQueue` — it is an `ObjectCounter` (on-ramp / off-ramp) plus the executor that drains it. The on-ramp counter increments on every accepted enqueue and decrements after the handler returns. The "capacity" is a long compared against the counter; "unprocessed-task count" is the counter's current value. Both are exposed on `TaskScheduler`: [`getUnprocessedTaskCount`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/TaskScheduler.java#L186-L201), [`getCapacity`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/TaskScheduler.java#L214-L220). The health monitor compares them — that comparison is the unhealthy-duration signal exercised in `pass1-04-event-creation-under-stress`.

Three put-paths exist on `InputWire`:

- **`put(data)`** (anchored at [`InputWire.put`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/wires/input/InputWire.java#L66-L73)) — calls `taskSchedulerInput.put`, which on a `SequentialTaskScheduler` calls `onRamp.onRamp()` (anchored at [`SequentialTaskScheduler.put`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/internal/SequentialTaskScheduler.java#L86-L91)). If the on-ramp counter is at capacity, the call blocks the producing thread until capacity becomes available. Same shape on `ConcurrentTaskScheduler.put` ([line 79–83](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/internal/ConcurrentTaskScheduler.java#L79-L83)).
- **`offer(data)`** (anchored at [`InputWire.offer`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/wires/input/InputWire.java#L75-L84)) — calls `attemptOnRamp`, which returns `true` if the increment succeeded and `false` if it would push over capacity. The producer sees the boolean; the data is silently dropped on `false`. Anchored at [`SequentialTaskScheduler.offer`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/internal/SequentialTaskScheduler.java#L93-L103).
- **`inject(data)`** (anchored at [`InputWire.inject`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/wires/input/InputWire.java#L86-L95)) — calls `forceOnRamp`, which increments unconditionally even past capacity. Used for control signals that must reach the handler even when the data queue is full; the topic file shows the `INJECT` solder on `EventDeduplicator::clear` and `EventDeduplicator::setEventWindow` in [`DefaultEventIntakeModule.initialize` (lines 109–114)](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-event-intake-impl/src/main/java/org/hiero/consensus/event/intake/impl/DefaultEventIntakeModule.java#L106-L114). The producer never blocks and never drops.

The structural reason `INJECT` is the right choice for those two control edges is cyclic backpressure. The deduplicator's `setEventWindow` carries the ancient-threshold update that the deduplicator itself uses to evict old descriptors and free space in its data structures — see the deduplicator's evict-on-event-window behaviour in [event-intake.md, "3. Deduplication"](../../architecture/topics/event-intake.md#3-deduplication). Tracing the wiring with the deduplicator at capacity: a `PUT` solder on `setEventWindow` would block the upstream producer of event-window updates until the deduplicator's queue drains; the queue can only drain after eviction advances; eviction advances only after `setEventWindow` lands and is processed. The cycle deadlocks. The wiring framework's start-time validator `checkForCyclicalBackpressure` detects this shape and logs a warning, but it does not silently rewrite the edge — the wiring author is the one who flips one edge in the cycle from `PUT` to `INJECT`. `setEventWindow` is the natural edge to flip because its producer's natural rate is low and its arrival is what unsticks the data path. The same pattern appears between modules in `PlatformWiring.wire(...)` on the persisted-event → gossip and event-creator → event-intake edges for the same structural reason (per the [wiring-framework topic file's Backpressure section](../../architecture/topics/wiring-framework.md#backpressure-wire-level)).

> **Load-bearing line.** *The capacity is one number per scheduler; every input wire on the same scheduler shares the same on-ramp counter; the three put-paths differ in what they do *when the counter is at capacity*.* This is the structural fact behind the misconception "the component owns its queue" and the misconception "`put` always blocks." The shared counter is what makes `c0-02-soldering-and-wires`'s edge-level `PUT`/`OFFER`/`INJECT` choice an edge-level choice rather than a per-wire-queue choice — and what makes `c0-03-backpressure-and-health-monitor-mechanics`'s health-monitor reading one duration per scheduler the correct shape rather than one per wire.

A configuration knob worth pinning down: `withExternalBackPressure(true)` declares that the producer applies backpressure externally and forces blocking insertion even when no capacity limit is set. This is documented in `AbstractTaskSchedulerBuilder` (the `withExternalBackPressure` method at [line 187–191](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/builders/internal/AbstractTaskSchedulerBuilder.java#L186-L191), and the topic file's Backpressure section: "it does not change the unhealthy-duration calculation in the health monitor"). The lesson does not walk this in detail; it is here so the learner knows the knob exists and that it sits on the builder, not on the scheduler.

The default `unhandledTaskCapacity` on the builder is `1`, not unbounded (anchored at [`AbstractTaskSchedulerBuilder.unhandledTaskCapacity = 1`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/builders/internal/AbstractTaskSchedulerBuilder.java#L44)). Consensus-layer schedulers always override this via `TaskSchedulerConfiguration` — see the orphan buffer's capacity 500 in [event-intake.md "Orphan buffer (topological ordering)"](../../architecture/topics/event-intake.md#5-orphan-buffer-topological-ordering) — so the default is effectively never used in production. The point of pinning the default is that absent configuration is *not* "no limit": it is "1," a deliberately low number that surfaces a wiring mistake fast.

`TaskSchedulerConfiguration` itself encodes its `unhandledTaskCapacity` field as `@Nullable Long` (anchored at [`TaskSchedulerConfiguration` line 29](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/builders/TaskSchedulerConfiguration.java#L27-L34)) with the convention that a non-null value is the capacity, a zero value is "unbounded," and a null value leaves the builder default unchanged. The two-layer convention — builder field default `1`, configuration null-means-leave-default, configuration zero-means-unbounded — is one of those places where the topic file and the code subtly differ in emphasis: the topic file says "the default is 1," and the configuration javadoc says "if null then 0 is used." Both are right against their own layer. The lesson treats the builder's `1` as the authoritative default because that is what the runtime instance sees when no configuration overrides it (anchored at the `configure` method's null-guard at [`AbstractTaskSchedulerBuilder.configure` line 102–104](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/builders/internal/AbstractTaskSchedulerBuilder.java#L102-L104), which only sets the capacity when the configuration's value is non-null).

### Chunk 4 — The canonical assembly pattern {#chunk-assembly-pattern}

`moment_id: assembly-pattern` (engagement-move inventory below)

Consensus-layer modules assemble each component in three steps inside their `initialize` method. The pattern is uniform across modules; `DefaultEventIntakeModule` is the representative example because every Pass 2 lesson under cluster A.2 will rely on it.

**Step 1 — construct a `ComponentWiring` per component.** The constructor takes the wiring model, the component's *interface* class (used for reflective method-reference resolution), and a `TaskSchedulerConfiguration` read from the module's typed `*WiringConfig`. The `ComponentWiring` constructor builds the underlying `TaskScheduler` immediately (see [`ComponentWiring` constructor line 146–151](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/component/ComponentWiring.java#L146-L151), which calls `model.schedulerBuilder(name).configure(schedulerConfiguration)...build()`). The scheduler exists from this point on; the component does not.

A line from `DefaultEventIntakeModule`, anchored at [`DefaultEventIntakeModule.initialize` line 96–97](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-event-intake-impl/src/main/java/org/hiero/consensus/event/intake/impl/DefaultEventIntakeModule.java#L96-L97):

```java
this.eventDeduplicatorWiring =
        new ComponentWiring<>(model, EventDeduplicator.class, wiringConfig.eventDeduplicator());
```

The `EventDeduplicator.class` argument is the *interface*; `ComponentWiring`'s constructor checks this at line 153–155 and throws if it isn't. The reason is reflective method-reference handling in `getInputWire` (see step 2): the wiring captures method references like `EventDeduplicator::handleEvent` and resolves them against the interface at wire-build time, before the implementation instance exists. The interface check at construction is what makes the deferred binding in step 3 type-safe.

**Step 2 — solder the component's input wires to its neighbours.** Anchored at [`DefaultEventIntakeModule.initialize` lines 103–131](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-event-intake-impl/src/main/java/org/hiero/consensus/event/intake/impl/DefaultEventIntakeModule.java#L103-L131). The mechanics of soldering are this cluster's next lesson (`c0-02-soldering-and-wires`); for `c0-01` what matters is that this step runs *after* the schedulers exist (step 1 built them) and *before* the component instances exist (step 3 binds them). At this point in the assembly, the scheduler holds method-reference stubs; calls to those stubs will route to the not-yet-bound handler once binding completes.

**Step 3 — bind the component instance.** Anchored at [`DefaultEventIntakeModule.initialize` lines 164–175](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-event-intake-impl/src/main/java/org/hiero/consensus/event/intake/impl/DefaultEventIntakeModule.java#L164-L175):

```java
final EventDeduplicator eventDeduplicator = new StandardEventDeduplicator(metrics, intakeEventCounter);
eventDeduplicatorWiring.bind(eventDeduplicator);
```

`ComponentWiring.bind(component)` walks the input-wire and transformer registrations (the lists `inputsToBind`, `transformersToBind`, `filtersToBind` populated by step 2) and resolves each captured method reference against the actual instance. Anchored at [`ComponentWiring.bind` line 662](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/component/ComponentWiring.java#L662). From this point the wiring is live: data flowing on a solder reaches the component's actual handler.

> **Load-bearing line.** *The scheduler is built at step 1 (so its on-ramp counter, capacity, primary output wire, and type are all set), the wiring graph is built at step 2 (so soldering can validate against scheduler types), and the component instance is bound last (so the wiring graph can be built before the implementation exists).* This ordering is what makes inter-module wiring work: `PlatformWiring.wire(...)` solders outputs of components from one module to inputs of components from another, but the schedulers and input-wire stubs on both sides exist after each module's step 1, well before any module's step 3.

A divergence from this pattern is worth noting because it surfaces in the topic file: `GossipWiring` does not use `ComponentWiring`. Per the wiring-framework topic (in [the wiring-framework topic file's TaskScheduler section](../../architecture/topics/wiring-framework.md#taskscheduler--taskschedulertype)), "`GossipWiring` self-builds its scheduler with its own `GossipWiringConfig` (treat the latter as an exception, not a template)." For a learner reading this lesson — the canonical template is `ComponentWiring`; gossip is one explicit deviation.

### Chunk 5 — Exception handling on a scheduler {#chunk-exception-handling}

The default `UncaughtExceptionHandler` installed when none is supplied to the builder is `ExceptionHandlers.defaultExceptionHandler(name)` — a `DefaultUncaughtExceptionHandler` record that logs at ERROR via log4j (anchored at [`ExceptionHandlers.DefaultUncaughtExceptionHandler.uncaughtException`](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/ExceptionHandlers.java#L75-L82)). The fall-back is wired in `AbstractTaskSchedulerBuilder` via `uncaughtExceptionHandlerOr(ExceptionHandlers.defaultExceptionHandler(name))` (anchored at [`AbstractTaskSchedulerBuilder` lines 281, 290–292](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/builders/internal/AbstractTaskSchedulerBuilder.java#L281-L292)).

The scheduler continues with the next task after the handler runs. The structural reason is in `SequentialTask.onExecute`, anchored at [lines 74–87](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-component-framework/src/main/java/com/swirlds/component/framework/schedulers/internal/SequentialTask.java#L70-L96): the `try { handler.accept(data); } finally { offRamp.offRamp(); ... nextTask.send(); }` shape always decrements the off-ramp counter and always sends the next task, regardless of what happened inside the handler. If an exception escapes the handler, it is delivered to `onException(t)` (line 92–95) which calls the uncaught exception handler. The next task in the chain has already been made eligible by the `nextTask.send()` in the finally block; the scheduler does not stall on the exception.

This matters because it shapes the contract a component handler must satisfy: a thrown exception is logged and the next task runs, but the off-ramp counter still decrements, so the on-ramp counter sees the failed task as drained. A handler that needs to retain a half-processed item on failure must catch the exception itself and re-enqueue (or drop, or quarantine). The framework's default is "log and continue"; component-level recovery semantics are the component's responsibility.

For `DIRECT` and `DIRECT_THREADSAFE` schedulers — no queue, no off-ramp counter — the handler is invoked on the caller's thread, and an exception that escapes the handler still reaches the same `UncaughtExceptionHandler` path. This is reflected in the topic file's note that those scheduler types "having no queue, invoke the handler on the calling thread but otherwise behave the same" with respect to exception handling.

### Moment `scheduler-spine` — at the end of Chunk 1

This is where the threshold concept lands: the scheduler is the locus of concurrency and queueing — not the component, not the wire. The learner is most likely to mis-import here, because the locus differs from servlet pools, actor libraries, and typical executor services.

### Moment `queue-mechanics` — at the end of Chunk 3

This moment cements the per-scheduler-total claim. It comes immediately after the load-bearing-line note that the capacity is one number per scheduler and asks the learner to derive a structural consequence.

**Move A — worked example with self-explanation.** Diagnosis tag: the learner is following Chunk 3's mechanism but may not have integrated the per-scheduler-total fact with the three put-paths. The worked example uses the deduplicator's three input wires concretely.

- Walk the worked example: the `EventDeduplicator`'s three input wires — `handleEvent` (data path), `setEventWindow` (control, INJECT solder), `clear` (control, INJECT solder) — and what happens at the scheduler's on-ramp counter when each fires under different load conditions. The load-bearing lines:
  - The `handleEvent` solder is `PUT` (anchored at [`DefaultEventIntakeModule` line 108](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-event-intake-impl/src/main/java/org/hiero/consensus/event/intake/impl/DefaultEventIntakeModule.java#L106-L108)) — *normal data; backpressure applies*.

- Self-explanation prompt at the second load-bearing line (verbatim):
  > The setEventWindow solder uses INJECT, and the on-ramp counter is shared across all three input wires on the deduplicator's scheduler. What invariant about the system would break if setEventWindow used PUT instead, when the deduplicator's scheduler is already at capacity from handleEvent traffic?
- `canonical_answer`: PUT blocks the producing thread when the on-ramp counter is at capacity. If setEventWindow used PUT, the upstream component publishing the event window — running on its own scheduler — would block waiting for the deduplicator's queue to drain. Event-window updates carry the ancient-threshold advance that the *deduplicator itself* uses to evict old descriptors and free space. So the deduplicator's queue would only drain after the producer of setEventWindow had blocked waiting for the deduplicator's queue to drain — a deadlock through a cycle in the wiring graph. INJECT breaks the cycle by letting the control signal land regardless of capacity; the on-ramp counter goes over capacity briefly while the event-window update is processed, and the eviction it drives is what brings the counter back below capacity.
- `alternative_correct_answers`:
  - An answer that frames the invariant as "control signals that drive eviction cannot be backpressured against the data they would evict" is correct at the structural altitude and stronger than the cycle-specific framing.
  - An answer that names the cycle without identifying eviction as the load-bearing mechanism is correct — the cycle alone justifies INJECT, eviction is the codebase-specific instance.
  - An answer that names `checkForCyclicalBackpressure` (the wiring-model validation pass at start time, anchored in `StandardWiringModel.start`) as the place that would warn about the cycle is correct as additional detail — the validator detects the shape; INJECT is the resolution.
- `followup` (verbatim, delivered when the learner names the cycle but treats it as "the framework prevents this" rather than "the wiring author chose INJECT to break the cycle"):
  > You named the cycle. But the framework does not silently rewrite PUT to INJECT — `checkForCyclicalBackpressure` logs a warning and the cycle stays if the wiring author leaves it. What is the human-level decision the wiring author makes at the moment of choosing INJECT for setEventWindow, and what is the cost of getting it wrong in either direction?
- `followup_canonical_answer`: the wiring author looks at each edge that is part of a cycle and chooses one to flip from PUT to INJECT — the choice is which edge can safely bypass backpressure. The cost of leaving it PUT (and thus leaving the cycle live) is the deadlock just described. The cost of flipping the wrong edge to INJECT is that the chosen edge no longer participates in backpressure: a fast producer on that edge can overrun the consumer's queue without ever blocking, so the on-ramp counter can grow unboundedly until something else (the health monitor signal, or in extreme cases the JVM running out of memory) intervenes. INJECT is the right answer for control signals where the producer's natural rate is low and the dead-locked alternative is worse; it is a wrong answer for high-volume data paths where backpressure is the only thing keeping the queue bounded.

- Cued-check prompt (verbatim):
  > A scheduler has three input wires bound to it. Each input wire receives one item simultaneously, and the scheduler's capacity is 2. The first two arrivals are PUT, the third is also PUT. ### Moment `assembly-pattern` — at the end of Chunk 4

## Contrasting cases material

This material supports the contrasting-cases move at moment `scheduler-spine`. It is referenced from the Engagement moves section above.

Three cases, each one a design where queue, threading policy, and (in some) an outbound channel are bundled into a unit:

**Case 1 — This codebase's `TaskScheduler`.**
A `TaskScheduler` instance owns a queue (on-ramp counter, capacity), a thread-execution policy (one of six `TaskSchedulerType` values), and a primary `OutputWire` that downstream consumers solder to. The component — the business-logic object with the handler methods — is a *separate* object that binds into the scheduler via `ComponentWiring.bind(component)`. Multiple input wires can be attached to the same scheduler; they share its queue.

Surface differences: the type-selection enum has six values, not the binary "one thread vs. pool" split typical of executor APIs. The component is bindable late (after construction and soldering), which lets the wiring graph be assembled before any component exists. The primary output wire is built into the scheduler — handlers that return a value forward it onto the wire, and downstream code reads from it; the wiring is therefore "structured" rather than "event-bus" or "queue-only."

**Case 2 — A textbook actor (Akka-style, Erlang-style).**
Each actor instance has one mailbox (its queue) and one logical thread of execution that processes messages from the mailbox in order. Threading is bundled into the actor; the actor and the mailbox are inseparable. Outbound is "send to another actor's mailbox" — there is no notion of a "primary output wire" because the receiving actor is named explicitly at send time.

Surface differences: the unit is the actor itself, not a separate scheduler-plus-component pair. Backpressure is typically not native — mailboxes are unbounded by default in most actor libraries, and bounding requires opting in to a mailbox type. There is no graph-walk validation; the system is built incrementally at runtime by actors creating and sending to other actors.

**Case 3 — A Java `ExecutorService` with a `BlockingQueue`.**
The executor owns a queue (the `BlockingQueue` passed to its constructor) and a thread pool with a fixed or dynamic size. Outbound is whatever the submitted task returns (a `Future`), or nothing if the task is a `Runnable`. There is no notion of a typed output wire; submission and result-handling are decoupled.

Surface differences: the queue is concretely a `BlockingQueue` (an in-memory queue with `put`/`offer`/`take`/`poll` operations and configurable bounding); the threading policy is "pool size and rejection policy." Multiple producers can call `execute` on the same executor; they share the queue. There is no notion of "input wires" — submission is method-call style, not graph-edge style.

**The deep invariant.**
All three designs tie one queue to one thread-execution policy to form a single concurrency-and-queueing unit. The unit's name differs (actor, executor, scheduler), and the unit's outbound shape differs (none, future-based, or typed-output-wire), but the invariant survives: the queue and the policy are inseparable, and multiple inputs into the unit share the unit's policy. This is the invariant the contrasting-cases prompt is asking the learner to articulate, with the implication that on this codebase the unit is the `TaskScheduler` — the component is a separate object that binds into the unit, which is the design feature that lets the framework assemble the entire wiring graph before any component exists.
