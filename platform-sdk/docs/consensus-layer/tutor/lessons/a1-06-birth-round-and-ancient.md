# Birth round and the ancient threshold

## Prerequisites

> - **`a1-02-rounds-and-witnesses`** — you can keep the three round quantities apart: **birth round** (stamped at creation, immutable, the key the DAG retains events by), **round-created** (the computed, mutable voting round), and **round-received** (set once at consensus). You also know the **pending consensus round** is the latest decided round + 1, and that events older than the ancient threshold are marked *terminal* (`ROUND_NEGATIVE_INFINITY`) and skipped by the algorithm.
> - **`a1-05-judges-and-consensus-order`** — you know that when a round decides it emits a **`ConsensusRound`** carrying an **`EventWindow`** for the round (the new ancient and expired thresholds and the next pending round), and that a round leaving consensus is what advances the window forward.

Beyond that, this lesson assumes `a1-01`'s picture of the hashgraph as a DAG of immutable events linked by self-parent and other-parent edges, and `a1-03`'s **strongly-seeing** as a black box — "an event reaches consensus only once enough later events strongly see the witnesses below it." This lesson is about the boundary of that DAG: which events are allowed *in*, which ones age *out*, and the one rule on birth round that keeps the aging-out from stranding events the algorithm still needs.

## Incoming retrieval probes

This is the sixth mechanism lesson in Cluster A.1. Three things from earlier lessons resurface; the tutor should consolidate against them when they come up — these are authorial watch-for signals, not an opening quiz.

- **Birth round as the creation-stamped retention key**, from `a1-02-rounds-and-witnesses` (and `a1-01`). Canonical statement the tutor consolidates against: *birth round is stamped by the creator at creation, is immutable, and is used for windowed DAG retention (ancient/expired filtering) and roster lookup — it is not the round the algorithm uses to order events.* The whole lesson runs on this quantity, so a firm grip on "birth round, not round-created" keeps the gate legible.
- **The `EventWindow` a decided round emits**, from `a1-05-judges-and-consensus-order`. Canonical statement: *a decided round produces a `ConsensusRound` carrying an `EventWindow` with the new ancient and expired thresholds and the next pending round; a round reaching consensus is what moves the window.* This lesson cashes that — the window's advance is exactly what sweeps the ancient threshold up. Consolidate explicitly if the learner treats the threshold as a fixed configuration value rather than something a decided round produces.
- **Pending consensus round**, from `a1-02-rounds-and-witnesses`. Canonical statement: *the pending consensus round is the latest decided round + 1, and a new event takes it as its birth round.* The future side of the gate turns on this, so watch for it and consolidate before the buffer material.

## Misconception watchlist

- **"The ancient threshold tests round-created (or round-received)."** *(Round-conflation — the recurring A.1 trap, here applied to filtering.)* Sounds like: "an event goes ancient when its voting round falls behind," "ancient is about where it reached consensus." Correction, in line: the ancient filter tests **birth round** and nothing else — `EventWindow.isAncient(event)` is exactly `event.getBirthRound() < ancientThreshold`. Round-created and round-received never enter the ancient/expired decision ([birth-round.md](../../concepts/birth-round.md), [`EventWindow.isAncient` #L86-L87](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-model/src/main/java/org/hiero/consensus/model/hashgraph/EventWindow.java#L86-L87)).
- **"Ancient means gone everywhere."** *(Over-generalization — collapsing two thresholds into one.)* Sounds like: "once it's ancient it's deleted," using *ancient* and *expired* interchangeably. Correction: ancient events are dropped from the **hashgraph DAG** but are still retained by **gossip**, which serves them to peers a few rounds behind; an event is discarded everywhere only once it passes the lower **expired** threshold. Ancient first, expired later ([event-lifecycle.md](../../concepts/event-lifecycle.md), [glossary: Ancient](../../glossary.md#ancient), [glossary: Expired](../../glossary.md#expired)).
- **"Ancient and stale are the same thing."** *(Over-generalization — conflating a state with a fate.)* Sounds like: "the ancient events are the stale ones." Correction: most events that go ancient *did* reach consensus and are not stale. **Stale** is the specific fate of an event that was admitted and then aged past the ancient threshold **without ever reaching consensus**; it is not a lifecycle state. (Stale mechanics are `a1-07`; here you only need the distinction.) ([glossary: Stale](../../glossary.md#stale), [stale-events.md](../../concepts/stale-events.md)).
- **"Ancient-ness is computed from generation."** *(Adjacent import — the hashgraph paper, and this codebase's own legacy.)* Sounds like: "isn't ancient based on generation, max parent generation + 1?" Correction: the **paper** uses a deterministic generation for the ancient horizon; **current code uses birth round**. A non-deterministic generation (`NGen`) is still computed locally, but only for topological ordering and "higher in the hashgraph" comparisons — never for the ancient/expired decision ([glossary: Generation](../../glossary.md#generation), [birth-round.md](../../concepts/birth-round.md), [hashgraph.md](../../architecture/topics/hashgraph.md)).
- **"A future event is rejected."** *(Over-generalization from the ancient drop.)* Sounds like: "an event from a round we haven't reached yet is thrown out." Correction: a **future** event — birth round above the pending consensus round — is **buffered**, not dropped, and released when the window advances to reach it. Only **ancient** events are dropped outright ([`FutureEventBuffer.addEvent` #L82-L97](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-utility/src/main/java/org/hiero/consensus/event/FutureEventBuffer.java#L82-L97)).

## Mechanism

Cluster A.1 has so far run the consensus algorithm *over* a DAG of events without asking which events are in the DAG to begin with. This lesson supplies that boundary. Birth round is the key on which it turns: it decides what enters the DAG, what stays, and — through one invariant — whether the boundary can move without breaking the algorithm above it ([hashgraph.md](../../architecture/topics/hashgraph.md#birth-round-filtering), [birth-round.md](../../concepts/birth-round.md)). The spine has four chunks: what birth round is, the three-way gate it drives, how the ancient threshold advances and what aging-out does, and the parent invariant that keeps the advance safe.

**Pre-training — terms this lesson integrates** (definitions live in the glossary and concept files; named here to set vocabulary):

- **Birth round** — the round a creator stamps on an event at creation, equal to its pending consensus round; immutable, and the key for ancient/expired filtering and roster lookup ([glossary: Birth round](../../glossary.md#birth-round), [birth-round.md](../../concepts/birth-round.md)).
- **Pending consensus round** — the round the node is currently working to decide, the latest decided round + 1; the birth round a new event receives ([glossary: Pending consensus round](../../glossary.md#pending-consensus-round)).
- **Event window** — the `EventWindow` record carrying `latestConsensusRound`, `newEventBirthRound`, `ancientThreshold`, and `expiredThreshold`; a decided round emits a new one ([glossary: Event window](../../glossary.md#event-window), [`EventWindow` #L20-L21](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-model/src/main/java/org/hiero/consensus/model/hashgraph/EventWindow.java#L20-L21)).
- **Ancient threshold** — the birth-round boundary on the window below which an event is ancient: dropped from the hashgraph DAG ([glossary: Ancient threshold](../../glossary.md#ancient-threshold)).
- **Expired threshold** — the lower birth-round boundary below which an event is expired: discarded everywhere, including by gossip. Mentioned here as the next rung down; its mechanics live outside the hashgraph (gossip retention) and are out of scope for this lesson ([glossary: Expired threshold](../../glossary.md#expired-threshold), [event-lifecycle.md](../../concepts/event-lifecycle.md)).
- **Future event** — an event whose birth round is above the pending consensus round; held in a **future-event buffer** and released when the window advances ([`FutureEventBuffer` #L22-L31](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-utility/src/main/java/org/hiero/consensus/event/FutureEventBuffer.java#L22-L31)).
- **Generation** — the paper-era key for the ancient horizon (max parent generation + 1); named here only to contrast with birth round ([glossary: Generation](../../glossary.md#generation)).
- **Stale** — the fate of an admitted event that aged past the ancient threshold without ever reaching consensus; the boundary this lesson stops at and `a1-07` takes up ([glossary: Stale](../../glossary.md#stale)).

### Chunk 1 — Birth round: the creation-stamped retention key `{moment: m1-birthround}`

Of the three round quantities `a1-02` kept apart, **birth round** is the one this lesson runs on. A creator stamps it on an event at creation, and it equals the creator's **pending consensus round** at that moment — the latest decided round + 1, the round the creator is working to reach consensus on ([birth-round.md](../../concepts/birth-round.md), [`EventWindow.getPendingConsensusRound` #L76-L77](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-model/src/main/java/org/hiero/consensus/model/hashgraph/EventWindow.java#L76-L77)). Birth round is **immutable** — it travels unchanged with the event through gossip, intake, and consensus — and is read off the event by `PlatformEvent.getBirthRound()` ([#L266-L267](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-model/src/main/java/org/hiero/consensus/model/event/PlatformEvent.java#L266-L267); the undefined sentinel is `EventConstants.BIRTH_ROUND_UNDEFINED = -1`, [#L22-L23](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-model/src/main/java/org/hiero/consensus/model/event/EventConstants.java#L22-L23)).

What birth round is *for* is the boundary of the DAG: it decides which events are admitted and how long they are retained, and it selects the roster an event is validated against ([birth-round.md](../../concepts/birth-round.md)). What it is *not* for is ordering — the algorithm orders events by round-created and round-received, never by birth round ([glossary: Birth round](../../glossary.md#birth-round)).

**Load-bearing line (signal):** the ancient/expired filter reads **birth round and nothing else**. Round-created (mutable) and round-received (set at consensus) never enter the filtering decision — keeping that straight is what makes the rest of the lesson legible.

### Chunk 2 — The three-way gate: future, admitted, ancient `{moment: m2-gate}`

Every event entering the consensus engine is sorted by its birth round into one of three fates, against two boundaries carried on the current `EventWindow`: the **pending consensus round** above and the **ancient threshold** below ([hashgraph.md](../../architecture/topics/hashgraph.md#birth-round-filtering)). `DefaultConsensusEngine.addEvent` runs the sort in two steps ([#L104-L137](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L104-L137)):

1. **`futureEventBuffer.addEvent(event)`** ([#L117](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L117)). This buffer is configured with `FutureEventBufferingOption.PENDING_CONSENSUS_ROUND`, so its releasable ceiling is `EventWindow.getPendingConsensusRound()` ([#L83-L84](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L83-L84), [`FutureEventBufferingOption` #L39-L43](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-utility/src/main/java/org/hiero/consensus/event/FutureEventBufferingOption.java#L39-L43)). `addEvent` returns the event when its birth round is at or below that ceiling (present), returns `null` and **buffers** it when its birth round is above (future), and returns `null` and **discards** it when it is already ancient ([`FutureEventBuffer.addEvent` #L82-L97](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-utility/src/main/java/org/hiero/consensus/event/FutureEventBuffer.java#L82-L97)). A future event is held and re-fed later, when `updateEventWindow` releases it (Chunk 3); an ancient one is gone.
2. **`linker.linkEvent(event)`** ([#L133](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L133)). For an event that survived step 1, `ConsensusLinker.linkEvent` short-circuits to `null` when `eventWindow.isAncient(event)` — `event.getBirthRound() < ancientThreshold` — and otherwise links it to its already-linked, non-ancient parents and admits it to the DAG ([`ConsensusLinker.linkEvent` #L78-L97](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/linking/ConsensusLinker.java#L78-L97), [`EventWindow.isAncient` #L86-L87](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-model/src/main/java/org/hiero/consensus/model/hashgraph/EventWindow.java#L86-L87)). A linked event is stored in the linker's two maps, keyed for windowed retention by birth round ([#L45-L53](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/linking/ConsensusLinker.java#L45-L53)).

Worked shape (from [birth-round.md](../../concepts/birth-round.md)): the pending consensus round is 100 and the ancient threshold is 90. An event with birth round 99 or 100 is **admitted**. An event with birth round 89 is **ancient** — dropped, and reported stale if it never reached consensus. An event with birth round 101 is **future** — it sits in the buffer until the window advances enough that the pending round is 101, then it is released into the gate.

**Structural detail (grounded in code, not paraphrase):** the two boundaries are different `EventWindow` quantities. The future ceiling is `getPendingConsensusRound()`, which is exactly `latestConsensusRound + 1` ([#L76-L77](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-model/src/main/java/org/hiero/consensus/model/hashgraph/EventWindow.java#L76-L77)); the ancient floor is `ancientThreshold`, an independent field on the same record. Intake has already applied its own birth-round ancient filter upstream, so most ancient events never reach the engine ([event-intake.md](../../architecture/topics/event-intake.md#birth-round-filtering)); the linker re-checks `isAncient` regardless, because the ancient threshold can advance partway through a single `addEvent` call — each decided round shifts the window mid-loop and the loop keeps processing events against the moved window (Chunk 3).

**Load-bearing line (signal):** **buffered is not dropped.** Future and ancient both make `addEvent` return `null` and both produce no consensus output from that call — but the future event is *retained for later* and the ancient event is *discarded*. The whole rest of the lesson is about events moving across the lower (ancient) boundary; the upper (future) boundary is a holding area, not a discard.

### Chunk 3 — How the ancient threshold advances, and what aging-out does `{moment: m3-advance}`

The ancient threshold is not a fixed setting — it is a field on the `EventWindow` that a **decided round** emits, and it sweeps upward each time a round reaches consensus (`a1-05`). When the engine sees a round decide, it takes that round's window and pushes it into the linker ([`DefaultConsensusEngine.addEvent` #L184-L193](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L184-L193)):

- **`linker.setEventWindow(eventWindow)`** ([#L188](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L188)) shifts the linker's birth-round-keyed window to the new `ancientThreshold` and, for every event that just fell below it, does three things: removes it from the hash lookup map, calls **`event.clear()`** on it, and adds it to a list of newly-ancient events that it returns ([`ConsensusLinker.setEventWindow` #L105-L116](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/linking/ConsensusLinker.java#L105-L116)).
- **The engine splits the returned list** ([#L189-L192](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L189-L192)): an event that already reached consensus is simply released; an event that aged out **without** reaching consensus is routed to the **stale-events output**. That is the boundary this lesson stops at — what stale means and what the receiver does with it is `a1-07` ([stale-events.md](../../concepts/stale-events.md)).
- **`futureEventBuffer.updateEventWindow(eventWindow)`** ([#L193](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L193)) shifts the *future* buffer to the new window, releasing any buffered events whose birth round is no longer ahead of the pending round; they are appended to the work queue and re-fed through `linkEvent` ([`FutureEventBuffer.updateEventWindow` #L107-L127](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-utility/src/main/java/org/hiero/consensus/event/FutureEventBuffer.java#L107-L127)). One decided round can thus admit events that were future a moment ago — the same advance that evicts at the bottom releases at the top.

**Structural detail (grounded in code, not paraphrase):** `EventImpl.clear()` is the load-bearing step, and its purpose is garbage collection. Removing the event from the linker's maps drops the *linker's* references to it, but the evicted event still holds references to its own parents, and a chain of ancient events would keep each other alive. `clear()` "erases all references to other events within this event … so other events can be garbage collected, even if this one still has things pointing to it" — so the ancient DAG fragment becomes collectable even while something above still points into it ([`EventImpl.clear` #L398-L407](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/EventImpl.java#L398-L407)). (`ConsensusImpl` separately drops ancient entries from its own working set and clears their algorithm metadata; the DAG-level link-severing is the linker's job — [birth-round.md](../../concepts/birth-round.md).)

**Load-bearing line (signal):** the advance is one event window doing two jobs at once — **evicting** events that fell below the new ancient threshold and **releasing** events that are no longer in the future. The eviction is where stale events are born; the release is how future events finally enter.

### Chunk 4 — The parent invariant: why the sweep doesn't strand events `{moment: m4-parent-invariant}`

The ancient threshold sweeps upward and evicts everything below it. For that sweep to be safe, one rule on birth round must hold: **a child's birth round is greater than or equal to every one of its parents' birth rounds** ([birth-round.md](../../concepts/birth-round.md)). Without it, the boundary could split a parent from its children. Suppose a child had a *lower* birth round than a parent, and the threshold swept to a value between them: the child is evicted as ancient while the parent stays non-ancient. If every child of that parent met the same fate, the parent would be left with **no living descendants in the DAG**. And an event with no living descendants can never be ancestrally reached by future events — so it can never accumulate the strongly-seeing relationships it needs to reach consensus (`a1-03`). It would be **stranded**: non-ancient by its own birth round, but cut off from the graph and unable to advance ([birth-round.md](../../concepts/birth-round.md)).

The invariant is enforced **upstream of the hashgraph**, at the event creator — not inside the consensus engine, which simply assumes it. The creator runs incoming events through its **own** `FutureEventBuffer`, configured with `FutureEventBufferingOption.EVENT_BIRTH_ROUND` ([`DefaultEventCreationManager` #L116-L117](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-event-creator-impl/src/main/java/org/hiero/consensus/event/creator/impl/DefaultEventCreationManager.java#L116-L117)). Any event whose birth round is above the local node's *desired event birth round* — the `newEventBirthRound` the node would stamp on a new event right now — is held in that buffer and never registered with the creator, so the creator can never select it as a parent. Once the window advances enough that the buffered event is no longer ahead, it is released ([birth-round.md](../../concepts/birth-round.md), [`FutureEventBufferingOption.EVENT_BIRTH_ROUND` #L15-L18](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-utility/src/main/java/org/hiero/consensus/event/FutureEventBufferingOption.java#L15-L18)). This is a **different** buffer from the consensus engine's, which is keyed on `PENDING_CONSENSUS_ROUND` (Chunk 2): the two gate different stages with different thresholds, and the event-creator one is what guarantees the parent invariant on locally-created events ([birth-round.md](../../concepts/birth-round.md)).

This is also where birth round differs structurally from the paper's **generation**. Generation is *derived* from an event's parents — one plus the maximum parent generation — so it is monotonic along ancestry by construction; the paper can use it as the ancient horizon and never worry about stranding. Birth round is *stamped by the creator*, not derived from parents, so its monotonicity is a property that must be **enforced** rather than assumed — which is exactly what the creator's `EVENT_BIRTH_ROUND` buffer does ([rounds-and-witnesses.md](../../concepts/rounds-and-witnesses.md), [glossary: Generation](../../glossary.md#generation)).

**Load-bearing line (signal):** birth round is a **monotonically advancing admission key**, and the whole admit-retain-evict staircase rests on that monotonicity. The sweep is safe only because the parent invariant guarantees the boundary never passes a child before its parent — and because that guarantee is a creator-stamped property, the algorithm pushes its enforcement upstream rather than discovering violations after the fact. *(See the Contrasting cases material below, which the tutor draws on at this moment.)*

### Moment `m3-advance`

*Why load-bearing (exposition):* the window advance is new material on this codebase, and the `clear()` step is the subtle, easily-skipped part — a learner can describe eviction as "remove from the map" and miss that the algorithm deliberately severs the event's own outgoing links for the garbage collector. This is also where the stale boundary is drawn.

**Move A — worked example with self-explanation.** *Diagnosis tag:* the learner is new to `setEventWindow` and the `clear()` step (the common case here).
- Walk (exposition): a round decides and emits an `EventWindow` raising the ancient threshold from 90 to 95. The engine calls `linker.setEventWindow(window)`. The linker shifts its birth-round window to 95, and for each event now below it (birth rounds 90–94) it removes the event from the hash map, calls `event.clear()`, and adds it to the list it returns ([`setEventWindow` #L105-L116](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/linking/ConsensusLinker.java#L105-L116)). The engine then routes the evicted events that never reached consensus to the stale output ([`DefaultConsensusEngine.addEvent` #L188-L192](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L188-L192)). **Load-bearing line:** eviction is not just "forget the event" — it is "sever the event's own links," and that extra step is for the garbage collector.
- Self-explanation prompt (verbatim): "When an event ages out, `setEventWindow` doesn't just drop it from the lookup maps — it also calls `event.clear()`, which erases that event's references to its own parents. - `canonical_answer`: The map removal drops the *linker's* reference to the event, but the evicted event still holds references *to its parents*, and those parents may hold references to theirs — a chain of ancient events keeps each other alive, so the garbage collector cannot reclaim the fragment even after the linker lets go. `clear()` erases the event's outgoing references, so the ancient sub-DAG becomes collectable even while something above it still points in — which is exactly what the method's JavaDoc says it is for ([`EventImpl.clear` #L398-L407](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/EventImpl.java#L398-L407)).
- `alternative_correct_answers`:
  - "It breaks the event's links to its parents so the whole ancient fragment can be GC'd; without it, ancient events keep each other alive even after the linker drops them."
  - "Map removal only removes one pointer; clear() severs the event's own references so the chain of ancient events doesn't pin itself in memory."
- `followup` (if the learner restates that clear() removes references without explaining the GC consequence): "That's what it does — now say *why it matters*. If the evicted event kept pointing at its parents, what would stop the garbage collector from reclaiming the ancient part of the graph?"

### Moment `m4-parent-invariant`

*Why load-bearing (exposition):* the threshold-concept moment. The whole lesson reduces to "birth round is a monotonic admission key, and the sweep is safe only because of the parent invariant." The contrasting cases place birth round against the paper's generation, and the prediction surfaces the stranding failure the invariant prevents.

**Move A — contrasting cases with comparison prompt.** *Diagnosis tag:* threshold concept; transfer is the goal. Uses the cases in the Contrasting cases material below.
- Prompt (verbatim): "Compare two ways to define the 'ancient' horizon that prunes the DAG. (1) The hashgraph paper uses an event's *generation* — one plus the maximum of its parents' generations — and the horizon is a generation threshold. (2) Current code uses an event's *birth round* — stamped by the creator at creation — and the horizon is a birth-round threshold, `EventWindow.isAncient`. What is the same about the role these two numbers play, what differs about where each number comes from, and what must stay true of *either* number along a parent-to-child edge for the sweeping horizon to prune the DAG without stranding an event?"
- `canonical_answer`: **Same role:** both define a moving lower boundary; events below it are pruned from the DAG, so the algorithm only ever works over a bounded recent window. **Difference:** generation is *derived from the event's parents* (max + 1), so it is monotonic along ancestry automatically; birth round is *stamped by the creator* and is not computed from the parents, so its monotonicity is not automatic. **What must stay true of either:** the number must be **monotonic along every parent-to-child edge** — a child's value at least its parents' — because if a child could sit below a parent, the rising horizon could evict the child while the parent remained, splitting the parent from its descendants ([birth-round.md](../../concepts/birth-round.md), [rounds-and-witnesses.md](../../concepts/rounds-and-witnesses.md), [hashgraph.md](../../architecture/topics/hashgraph.md)).
- `alternative_correct_answers`:
  - "Both are a moving prune boundary; generation is computed from parents (monotonic for free), birth round is creator-stamped (monotonicity must be enforced); either way the key has to be non-decreasing from parent to child or the sweep strands events."
  - "Same: a lower horizon that bounds the working DAG. Different: derived-from-parents vs stamped-by-creator. Required: child ≥ parent along ancestry, so the horizon never passes a child before its parent."
  - "They play the identical role; the paper's number is intrinsic to the ancestry while birth round is assigned at creation; both must never decrease from a parent to its child."
- *Deep invariant (exposition for consolidation, not read as a question):* a sweeping admission horizon is safe only if the key it sweeps is monotonic along ancestry. Generation earns that monotonicity by construction (it is derived from the parents); birth round must have it enforced upstream, at the event creator's `EVENT_BIRTH_ROUND` buffer, because a creator-stamped number could otherwise be set below a parent's. The stranding of a parent with no living descendants — cut off from the strongly-seeing it needs to reach consensus — is the failure the enforcement exists to prevent.

**Move B — prediction-and-reveal.** *Diagnosis tag:* the learner is fluent on the ancient drop (Chunks 2–3) and `a1-03`'s strongly-seeing as a black box, and can reason about a hypothetical violation.
- `answer_shape`: a failure trace — *which event is evicted, which stays, and what becomes of the one that stays* — plus where the violation is prevented.
- Framing + prompt (verbatim): "Suppose, hypothetically, the network let a creator build a child event with a *lower* birth round than one of its parents. Now let the ancient threshold sweep up to a value sitting between the child's birth round and the parent's. Predict which of the two — child or parent — is evicted as ancient and which stays, then say what that does to the one that stays: can it still reach consensus? Finally, name where in the system a creator is stopped from ever building such a child in the first place."
- `canonical_answer`: The **child** (lower birth round) is evicted as ancient; the **parent** (higher birth round) stays non-ancient. If every child of that parent shared the fate, the parent is left with **no living descendants** in the DAG. An event with no living descendants can never be ancestrally reached by new events, so it can never accumulate the strongly-seeing relationships it needs to reach consensus — it is **stranded**: non-ancient by its own birth round, but cut off from the graph and unable to advance. That is the reason for the parent invariant (child birth round ≥ every parent's), and the event creator enforces it upstream: its `EVENT_BIRTH_ROUND` future-event buffer holds any event whose birth round is above the round the node would currently stamp, so such an event is never registered and can never be chosen as a parent ([birth-round.md](../../concepts/birth-round.md), [`DefaultEventCreationManager` #L116-L117](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-event-creator-impl/src/main/java/org/hiero/consensus/event/creator/impl/DefaultEventCreationManager.java#L116-L117)).
- `alternative_correct_answers`:
  - "Child evicted, parent kept; the parent loses all its descendants, so future events can't reach it and it never strongly-sees its way to consensus — stranded. Prevented at the creator's EVENT_BIRTH_ROUND buffer."
  - "The lower-birth-round child goes ancient while the parent lingers; a parent with no living descendants is cut off and can never reach consensus, which is why creators are stopped from picking a higher-birth-round event as a parent."
- `followup` (if the learner identifies the eviction but stops short of the consensus consequence): "Right on which one is evicted. Now push it one step: the parent is still non-ancient but has no descendants left. What does a hashgraph event *need* from later events to reach consensus, and can a parent with no living descendants ever get it?"

## Contrasting cases material

The threshold concept is *birth round is a monotonically advancing admission key, and the ancient sweep is safe only because the parent invariant keeps it from passing a child before its parent.* The two cases hold "a number that defines the ancient horizon" constant and vary where the number comes from.

- **Case 1 — generation horizon (the paper, and this codebase's legacy).** Each event's generation is one plus the maximum of its parents' generations; the ancient horizon is a generation threshold. Because generation is *derived from the parents*, a child's generation is always at least its parents' — monotonicity along ancestry is automatic, and the horizon can never evict a child while keeping a parent ([glossary: Generation](../../glossary.md#generation), [rounds-and-witnesses.md](../../concepts/rounds-and-witnesses.md)). *Surface:* the key is computed from ancestry; monotonicity is free. *Who guarantees it:* the construction.
- **Case 2 — birth-round horizon (current code).** Each event's birth round is stamped by its creator at creation (the pending consensus round); the ancient horizon is `EventWindow.isAncient`, a birth-round threshold ([`EventWindow.isAncient` #L86-L87](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-model/src/main/java/org/hiero/consensus/model/hashgraph/EventWindow.java#L86-L87)). Because birth round is *assigned*, not derived, nothing in the number itself forces a child above its parent; the parent invariant has to be enforced, which the creator does with its `EVENT_BIRTH_ROUND` future-event buffer ([birth-round.md](../../concepts/birth-round.md)). *Surface:* the key is stamped at creation; monotonicity is enforced. *Who guarantees it:* the event creator, upstream.

**The deep invariant that survives the surface differences:** in both cases the horizon is a sweeping lower boundary, and in both cases the boundary is safe only if the key it sweeps is **monotonic along ancestry** — a child never below a parent. The difference is purely in *how that monotonicity is obtained*: generation earns it by being derived from the parents, birth round earns it by upstream enforcement at the creator. The cost of the birth-round choice is that the guarantee lives outside the hashgraph and must be maintained; the benefit (developed in the delta material and later clusters) is that birth round ties an event to a *round* — and therefore a roster — rather than to a raw ancestry depth. *(Why generation is still computed locally as `NGen` — for topological ordering, not for the horizon — is `a1-02`/`a1-05` material; this case set is only about what defines the ancient boundary.)*

## Close-out retrieval

- Free-recall summary (verbatim): "In your own words: how does birth round decide an event's fate as it enters the consensus engine and as the ancient threshold later sweeps past it — and what one rule on birth round keeps that sweep from stranding an event the algorithm still needs?"
  - `canonical_answer`: On entry, an event is sorted by birth round into three fates against the event window — **future** (above the pending consensus round: buffered and released when the window advances), **admitted** (in-window: linked into the DAG), or **ancient** (below the ancient threshold: dropped). As rounds decide, the window's ancient threshold sweeps up; `setEventWindow` evicts the now-ancient events (severing their links for GC) and the engine routes those that never reached consensus to the stale output. The sweep stays safe because of the **parent birth-round invariant** — a child's birth round is never below a parent's — which the event creator enforces upstream with its `EVENT_BIRTH_ROUND` buffer, so the boundary can never evict a child while stranding its parent.
  - `alternative_correct_answers`:
    - "Birth round sorts entering events into future-buffered, admitted, or ancient-dropped; the ancient threshold rises as rounds decide, evicting old events (and emitting the non-consensus ones as stale); the parent invariant (child ≥ parent) keeps the sweep from stranding a parent, enforced at the creator's buffer."
    - "On the way in: future (held), in-window (linked), ancient (dropped). On the way out: the window advances, ancient events are evicted and cleared for GC, the never-consensus ones go stale. Safe because birth round is monotonic along ancestry, enforced upstream so no child is evicted ahead of its parent."
