# Judges and consensus order

## Prerequisites

> - **`a1-02-rounds-and-witnesses`** — you can distinguish **round-created** (the mutable *voting round*, recalculated as earlier rounds decide) from **round-received** (the *consensus round*, set once when an event reaches consensus and never changed afterward), and you know a round is **decided** once every witness in it has a fame verdict.
> - **`a1-04-fame-voting-and-coin-rounds`** — you can state that the product of a decided election is a set of **famous witnesses**, that a famous witness is *not yet* a judge, and that a branched creator may be voted famous on more than one of its witnesses in the same round.

Beyond that, this lesson assumes `a1-03`'s **branching** (one creator, two events sharing a self-parent) and **strong-seeing** (a super-majority of distinct creators' paths agreeing on a canonical witness) as background — they explain *why* a creator can leave more than one famous witness in a round, which is the case the judge merge has to handle. This lesson picks up exactly where `a1-04` stopped — at *famous* — and carries one decided round all the way to a totally-ordered list of consensus events.

## Incoming retrieval probes

This is the fifth mechanism lesson in Cluster A.1. Three things from `a1-02` and `a1-04` resurface; the tutor should consolidate against them when they come up — these are authorial watch-for signals, not an opening quiz.

- **A decided round**, from `a1-04-fame-voting-and-coin-rounds`. Canonical statement the tutor consolidates against: *a round is decided when every witness in it has a fame verdict — `RoundElections.isDecided()` — and its product is a set of famous witnesses.* The whole of this lesson runs downstream of that instant, so a firm grip here keeps the hand-off legible.
- **Famous witness is not yet a judge**, from `a1-04-fame-voting-and-coin-rounds`. Canonical statement: *fame is a per-witness yes/no verdict, and a branched creator can have more than one famous witness in a round.* This lesson cashes the forward-promise `a1-04` made — the collapse to one judge per creator is the first thing it teaches; consolidate explicitly if the learner conflates the two.
- **Round-created versus round-received**, from `a1-02-rounds-and-witnesses`. Canonical statement: *round-created is the mutable voting round and can be recalculated; round-received is set once at consensus and never changes.* The last chunk turns on this distinction, so watch for it resurfacing and consolidate before the INV-001 material.

## Misconception watchlist

- **"A famous witness is the same thing as a judge."** *(Over-generalization — collapsing the merge `a1-04` deferred.)* Sounds like: "so the famous witnesses fix the order," using the words interchangeably. Correction, in line: a judge is the *one* famous witness per creator after a deterministic merge; a branched creator can be voted famous on several witnesses but contributes exactly one judge, picked by minimum base hash ([judges.md](../../concepts/judges.md), [`findAllJudges` #L136-L164](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/RoundElections.java#L136-L164)).
- **"A leader or coordinator picks the consensus order."** *(Adjacent-protocol import — PBFT primary, Raft leader.)* Sounds like: "which node proposes the ordering?", "who is the round leader?" Correction: no node proposes anything. The order is *computed* from the decided graph by a fixed comparator; there is no proposer, no ordering message, and no view-change, because nothing was chosen to fail over ([hashgraph.md](../../architecture/topics/hashgraph.md), [`ConsensusSorter.compare` #L42-L75](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusSorter.java#L42-L75)).
- **"The whitening is a network-shared random value."** *(Adjacent-protocol import — common-coin BFT; or carry-over from `a1-04`'s coin field.)* Sounds like: "where does the round's whitening come from?", "is it the coin again?" Correction: the whitening is the **XOR of this round's judges' own base hashes** — a deterministic per-round byte string every node computes identically from the judge set, used only as the final tie-break in the sort. It is not random, not shared over the wire, and unrelated to the `a1-04` coin ([`generateWhitening` #L53-L67](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusUtils.java#L53-L67)).
- **"Round-received can change later, the way round-created does."** *(Over-generalization from `a1-02`'s recalculation.)* Sounds like: "if the roster changes, does the event's consensus round get recomputed too?" Correction: round-received is set once, the instant the event reaches consensus, and never changes; it is round-*created* that recalculateAndVote re-derives. The judges' metadata anchors that recalculation, which is the subject of the last chunk ([rounds-and-witnesses.md](../../concepts/rounds-and-witnesses.md), [`setIsConsensusTrue` #L834-L844](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusImpl.java#L834-L844)).

## Mechanism

`a1-04` ended at *famous*: every witness in the election round has a fame verdict, and the round is decided. This lesson is what the algorithm does next — turn that set of famous witnesses into a single total order over the events the round receives. The spine has four chunks: collapse the famous witnesses to judges, freeze the set, sort the round's events, and then see why the judges' metadata is load-bearing far beyond this one round ([hashgraph.md](../../architecture/topics/hashgraph.md), [judges.md](../../concepts/judges.md)).

**Pre-training — terms this lesson integrates** (definitions live in the glossary and concept files; named here to set vocabulary):

- **Judge** — the unique famous witness for a creator in a decided round; the judges of a round jointly fix the round-received of every event that is an ancestor of all of them ([glossary: Judge](../../glossary.md#judge), [judges.md](../../concepts/judges.md)).
- **Consensus order** — the total order over a decided round's events that the algorithm produces, fixed by the judges and split into per-transaction consensus timestamps ([glossary: Consensus order](../../glossary.md#consensus-order)).
- **Round-received** (= *consensus round*) — the round in which an event reaches consensus; set once and never changed ([glossary: Round-received](../../glossary.md#round-received)).
- **Whitening** — an XOR of the round's judges' base hashes, used as the final tie-break when sorting events into consensus order ([judges.md](../../concepts/judges.md), [`generateWhitening` #L53-L67](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusUtils.java#L53-L67)).
- **cGen (local consensus generation)** — a temporary, within-round topological generation assigned to a round's consensus events for use as a sort key, then cleared ([`LocalConsensusGeneration` #L15-L26](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/LocalConsensusGeneration.java#L15-L26)).
- **Preliminary consensus timestamp** — the median of the times the judges first saw an event; the first and primary sort key, later nudged strictly increasing ([glossary: Consensus timestamp](../../glossary.md#consensus-timestamp)).

### Chunk 1 — From famous witnesses to judges: the deterministic merge `{moment: m1-judges}`

Once `RoundElections.isDecided()` is true — `numUnknownFame == 0 && !elections.isEmpty()` — the round's famous witnesses are collapsed to **one judge per creator** by `RoundElections.findAllJudges` ([`isDecided` #L99-L101](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/RoundElections.java#L99-L101), [`findAllJudges` #L136-L164](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/RoundElections.java#L136-L164)). The collapse is a `Map<NodeId, EventImpl>` keyed by creator: each famous witness is merged into the map under its creator id, so a creator with one famous witness contributes it directly, and a creator with several contributes the result of the merge function ([#L142-L149](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/RoundElections.java#L142-L149)). The judges are then sorted by creator id and each is flagged `setJudgeTrue` ([#L154-L161](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/RoundElections.java#L154-L161), [`EventImpl.setJudgeTrue` #L162-L165](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/EventImpl.java#L162-L165)).

The only interesting case is a **branched** creator, which `a1-03` showed can leave more than one famous witness in a round ([branching.md](../../concepts/branching.md)). `RoundElections.uniqueFamous` resolves it by keeping the witness with the **lexicographically-least signed base hash** — `ArrayUtils.arrayCompare` over the two events' base-hash bytes ([`uniqueFamous` #L174-L187](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/RoundElections.java#L174-L187)). The judges are persisted as `JudgeId` records — creator id plus the judge's base hash — inside the round's `ConsensusSnapshot` ([`platform_state.proto` JudgeId #L168-L178](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/hapi/hedera-protobuf-java-api/src/main/proto/platform/state/platform_state.proto#L168-L178)).

**Load-bearing line (signal):** the merge key is the **creator** (one judge per creator), and the tie-break is the **event's hash** (a value identical on every node). Together these are why branching "never inflates a creator's voice in the consensus order" — a brancher still gets exactly one judge, and every honest node picks the same one without exchanging a message ([branching.md](../../concepts/branching.md)).

### Chunk 2 — When the round decides and the judge set freezes `{moment: m2-frozen}`

`findAllJudges` is called from `roundDecided`, and `roundDecided` is reached through one specific gate. Inside `calculateAndVote`, after voting, the engine checks `roundElections.isDecided() && !waitingForInitJudges()`; when that holds it calls `roundDecided` ([#L416-L424](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusImpl.java#L416-L424)). The comment at that gate is the load-bearing fact: once the round is decided, the next two rounds already have events in them (or fame could not have been decided), so **any witness added to this round in the future is instantly decided not-famous** — the set of famous witnesses, and therefore the judges, "is now completely known and immutable" ([#L417-L423](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusImpl.java#L417-L423)).

`roundDecided` then finds the judges, checks them, and fixes the round-received of every event that is an ancestor of all of them ([`roundDecided` #L706-L766](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusImpl.java#L706-L766)). Deciding one round can decide the next: a single `addEvent` call runs `recalculateAndVote` in a loop, appending a `ConsensusRound` for each round that decides until none remains ([`addEvent` #L304-L309](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusImpl.java#L304-L309)).

**Structural detail (grounded in code, not paraphrase):** `roundDecided` also runs a `checkJudges` guard that is pure observability, not enforcement — it logs an error if there are no judges, and logs an error if the judges carry **less than a super-majority of roster weight**, but it neither throws nor halts ([`checkJudges` #L774-L791](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusImpl.java#L774-L791)). The super-majority of weight on judges is a property the decided round is expected to have, surfaced as a log rather than asserted in the control flow.

### Chunk 3 — Judges to total order: whitening and the four-key sort `{moment: m3-sort}`

This is the heart of the lesson and the part that is new on this codebase, so it is worth working an example through. `roundDecided` computes the round's whitening once — `ConsensusUtils.generateWhitening(judges)` XORs every judge's base hash into a single `SIG_SIZE_BYTES`-wide byte string — and hands it to `findConsensusEvents` ([`generateWhitening` #L53-L67](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusUtils.java#L53-L67), [#L720-L721](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusImpl.java#L720-L721)).

`findConsensusEvents` gathers the **common ancestors of all the judges** that have not yet reached consensus, marks each consensus with `setIsConsensusTrue` (which sets its round-received and its preliminary consensus timestamp — the median of the times the judges first saw it), sorts them, then assigns each a consensus order number ([`findConsensusEvents` #L804-L825](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusImpl.java#L804-L825), [`setIsConsensusTrue` #L834-L844](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusImpl.java#L834-L844)). The sort itself, `ConsensusSorter.compare`, applies **four keys in this fixed order** ([`ConsensusSorter` #L29-L75](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusSorter.java#L29-L75)):

1. **Preliminary consensus timestamp** — the median of the received times ([#L46](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusSorter.java#L46)).
2. **Extended median of received times** — on a tie, compare the two events' received-time lists at the median position, then walk outward symmetrically (median−1, median+1, median−2, …) until they differ ([#L52-L64](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusSorter.java#L52-L64)).
3. **cGen** — a within-round topological generation assigned by `LocalConsensusGeneration.assignCGen`: events are visited in NGen order, and each event's cGen is one more than the maximum cGen of its parents that are also in this round's set (undefined parents count as 0) ([#L67](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusSorter.java#L67), [`assignCGen` #L36-L50](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/LocalConsensusGeneration.java#L36-L50)).
4. **Whitened base hash** — the two events' base hashes compared under the round's whitening ([#L73-L74](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusSorter.java#L73-L74)).

Finally `setConsensusOrder` walks the sorted list, nudging each event's timestamp strictly later than the previous one's so transactions never share a timestamp, and stamps the monotonic `numConsensus` counter on each ([`setConsensusOrder` #L856-L874](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusImpl.java#L856-L874)).

**Structural detail (grounded in code, not paraphrase):** every one of the four keys is a function of the shared graph and the judge set — the received times are when the judges first saw the event, cGen is a topological property of the round's events, and the whitening is the judges' own hashes XOR-ed together. None of them depends on a node's local clock or the order it received events over gossip, so two honest nodes with the same graph produce the **identical** total order. Note also that the `ConsensusSorter.compare` doc comment is stale — it lists "roundReceived, then consensusTimestamp, then generation, then whitened signature" and omits the extended-median key ([#L38-L41](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusSorter.java#L38-L41)); the implementation at `#L42-L75` is the authoritative four-key order, and the topic file flags the same staleness ([hashgraph.md](../../architecture/topics/hashgraph.md)).

### Chunk 4 — The judges as a load-bearing anchor: round-received, round-created, and INV-001 `{moment: m4-derived}`

The order this lesson just built is permanent: an event's round-received is set exactly once in `setIsConsensusTrue` and never changes, and once an event reaches consensus it leaves the working set and its round-created is no longer recalculated ([rounds-and-witnesses.md](../../concepts/rounds-and-witnesses.md), [`setIsConsensusTrue` #L834-L844](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusImpl.java#L834-L844)). But the judges keep doing work after their round decides. When a later round decides, `recalculateAndVote` clears the round-created of every event still in the working set — non-ancient and non-consensus — and re-derives it — this is the recomputation `a1-02` flagged, run in anticipation of roster changes ([`recalculateAndVote` #L325-L371](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusImpl.java#L325-L371)). To re-derive a round-created you need a fixed anchor at the bottom, and the **judges of the last decided round are that anchor**: a judge whose parents are all terminal (`ROUND_NEGATIVE_INFINITY`) keeps its metadata intact so descendants can compute their rounds against it ([#L330-L350](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusImpl.java#L330-L350)).

The carve-out is narrow on purpose, and the reason is **INV-001**: `roundCreated(child) >= roundCreated(parent)` along every ancestry chain, a definitional consequence of how the paper assigns rounds ([INV-001](../../invariants/INV-001-roundcreated-monotonic-along-ancestry.md)). The condition the code checks is not "is this a judge" but "is this a judge **and** are all its parents terminal" — i.e. it has no non-terminal ancestor. That extra clause is the fix for **SCN-001**: under the original rule, *every* last-round judge was exempt from clearing, and if two same-round judges `J1 → … → J2` shared an ancestry chain with a non-judge `A` between them, a roster change could recompute `A` up to round `N+1` while the exempt `J2` stayed at `N` — producing `roundCreated(A) = N+1 > roundCreated(J2) = N` with `A` an ancestor of `J2`, breaking INV-001 and stalling consensus network-wide ([SCN-001](../../scenarios/SCN-001-same-round-judge-ancestry-stalls-consensus.md)).

**Load-bearing line (signal):** the consensus order is **derived**, not chosen — it is a pure function of the decided graph through the judge set. That is the strength of the design (no coordinator to fail) and its one fragility: the derivation rests on the judges' metadata, so the algorithm must preserve exactly the judges that can safely anchor recalculation and clear the rest. *(See the Contrasting cases material, which the tutor draws on at this moment.)*

## Engagement moves

### Moment `m1-judges`

*Why load-bearing (exposition):* this is where `a1-04`'s deferred promise — that famous witnesses become judges — is finally cashed, and where the branched-creator merge (the only subtle case) is introduced. If the learner leaves still equating famous witnesses with judges, the rest of the lesson is mis-shaped.

**Move A — prediction-and-reveal.** *Diagnosis tag:* the learner is fluent on `a1-04`'s fame verdict and on `a1-03`'s branching, and can reason about what a fair ordering needs.
- `answer_shape`: a count per creator plus the *kind* of selection rule — *how many judges a creator gets, and that the tie-break must be a deterministic function of the events.*
- Framing + prompt (verbatim): "Last lesson left us with a round's famous witnesses, and you saw a branched creator can be voted famous on more than one of its witnesses in the same round. This lesson turns famous witnesses into *judges* — the events that fix consensus order. Before I show the merge: thinking about what a fair order needs, how many judges should a single creator contribute to a round, and if a branched creator left two famous witnesses behind, what kind of rule do all the nodes need to pick the same one without talking to each other?"
- Confidence elicitation (verbatim, optional): "How sure are you, low to high, before I reveal it?"
- `canonical_answer`: Exactly **one judge per creator** — otherwise a brancher would inflate its voice in the order — and the tie-break for a branched creator must be a **deterministic function of the events themselves**, so every node computes the same choice with no message. In the code that function is "keep the witness with the minimum base hash," but any node-independent function of fixed event data would do ([judges.md](../../concepts/judges.md), [branching.md](../../concepts/branching.md), [`uniqueFamous` #L174-L187](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/RoundElections.java#L174-L187)).
- `alternative_correct_answers`:
  - "One per creator; pick by something fixed on the events, like a hash, so it's the same everywhere — no coordination."
  - "Each creator gets a single judge; the branched case is broken by a deterministic rule over the events (e.g. least hash), not by a vote or by arrival order."
  - "One, so branching can't buy extra weight; the choice has to be a pure function of the two events so all honest nodes agree."
- `followup` (if the learner proposes a non-deterministic pick — earliest-arrived, first-seen, a vote): "Two nodes received the brancher's two witnesses in opposite orders. If the judge is 'the one that arrived first,' do they pick the same judge? What would you key the choice on instead so they always agree?"

**Move B — worked example with self-explanation.** *Diagnosis tag:* the learner is new to the merge and wants to see the branched case concretely.
- Walk (exposition): four creators A, B, C, D; A branched and was voted famous on two witnesses `a₁` and `a₂` in round 5, while B, C, D each have one famous witness. `findAllJudges` builds a map keyed by creator: B, C, D map straight in; A's two entries collide, so `uniqueFamous` runs and keeps whichever of `a₁`, `a₂` has the lexicographically-least base hash. ### Moment `m4-derived`

*Why load-bearing (exposition):* the threshold-concept moment. - `canonical_answer`: Both produce one total order on the batch that all honest nodes agree on.   - "Leader-based order is proposed and must be recovered if the leader dies; judge-derived order is recomputed identically everywhere, so there is no leader and no recovery path to need."

## Contrasting cases material

The threshold concept is *derived order: a decided round's consensus order is computed from the judges by a fixed function, not chosen by any node.* The two cases hold "agree on one total order for a batch" constant and vary how the order is produced.

- **Case 1 — leader-based BFT (PBFT / Raft).** An elected primary or leader proposes the order of the batch in a message; followers accept it; a value commits once a super-majority acknowledges. If the leader fails or is suspected, a view-change / election picks a new one before progress resumes ([voting.md](../../concepts/voting.md) for the super-majority parallel). *Surface:* an ordering message per batch, and a recovery protocol for leader failure. *Who decides the order:* a coordinator, by proposal.
- **Case 2 — hashgraph judge-derived order.** The round's judges are one famous witness per creator, fixed deterministically by the decided graph; the order of the round's events is the four-key sort, whitened by the judges' own hashes ([`ConsensusSorter.compare` #L42-L75](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusSorter.java#L42-L75)). *Surface:* no message, no proposer, no recovery protocol. *Who decides the order:* nobody — it is computed.

**The deep invariant that survives the surface differences:** both reach one agreed total order, but Case 1 makes the order an output of a *chosen* coordinator and therefore needs a failure-recovery path, while Case 2 makes the order a deterministic *function* of state every honest node already shares. Because there is no coordinator, there is nothing to fail over: the order's safety and liveness rest on the judge computation being reproducible, not on any node staying up. The cost is that the derivation's anchor — the judges' metadata — becomes load-bearing for everything downstream, which is why the algorithm guards how judges are preserved across roster changes (INV-001, SCN-001). *(How birth round decides which events are even eligible to enter this sort is `a1-06`. This case set is only about how a decided round's events are ordered and why no coordinator is involved.)*

The order is the same everywhere because every key is a deterministic function of the shared graph and the judge set, with no coordinator and no ordering message — it is derived, not chosen.
  - `alternative_correct_answers`:
    - "Merge to one judge per creator, the judges fix round-received for their common ancestors, then a fixed four-key sort orders them; it's deterministic from the graph, so no node has to propose it."
    - "Famous witnesses collapse to judges; judges' common ancestors take this round-received; the four-key whitened sort puts them in order — identical on every node because it's computed from the same graph, not chosen."
