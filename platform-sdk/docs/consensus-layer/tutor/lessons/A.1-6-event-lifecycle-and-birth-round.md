# Event lifecycle and birth round

## Where we are

This is the sixth lesson in cluster A.1 (Hashgraph Algorithm). [A.1-1 the hashgraph DAG](A.1-1-hashgraph-dag.md) noted that the in-memory DAG is the *non-ancient slice* of an unbounded historical event log and deferred the question of how the slice is bounded; [A.1-2 rounds and witnesses](A.1-2-rounds-and-witnesses.md) through [A.1-5 judges](A.1-5-judges.md) walked the algorithm that runs over the slice and ended on `roundDecided` packaging a `ConsensusRound` whose `EventWindow` carries the new ancient and expired thresholds the next round will run against. This lesson is where the bounding contract becomes mechanical: which events `ConsensusLinker` accepts, which it rejects up front, which it evicts when the window advances, and which retention sites outside the hashgraph still hold events the consensus algorithm has finished with. The next lesson, [A.1-7 stale events](A.1-7-stale-events.md), takes the post-admission slice of this same machinery — events that entered the DAG and aged out without reaching consensus — and walks the stale-event reporting that goes with it.

## Motivating problem

A consensus node cannot remember every event ever gossiped. Memory is finite, and the algorithm in A.1-2 through A.1-5 only needs the *current* round's witnesses, the witnesses of the small handful of rounds whose fame has not yet decided, and the events whose round-received computation those judge sets will eventually fix. Past that, retention is a different concern: gossip needs older events to serve peers who are catching up, and the signed-state pipeline needs the events of a round at the moment that round reaches consensus, but the consensus algorithm itself does not.

That gives two distinct horizons. The closer one is the *ancient* horizon: events older than this no longer affect ongoing consensus and are evicted from the hashgraph DAG. The farther one is the *expired* horizon: events older than this are dropped from the auxiliary retention sites — the gossip shadowgraph and the consensus-rounds buffer — that hold events for catch-up after the algorithm has finished with them. Between the two horizons is an operational window: a node has finished working on these events itself but will still serve them to a slower peer.

The other half of the problem is the *handle*. Every node has to agree, deterministically and Byzantine-tolerantly, on which side of each horizon a given event sits. A locally-computed property like NGen will not work — different nodes can compute different NGens for the same event, depending on which parents they have linked at the moment they compute it, and the algorithm cannot tolerate that disagreement at the lifecycle boundary. The handle has to be content-addressable and creator-signed: every node that receives an event reads the same number off it. That handle is *birth round* — the round the creator believed was pending at the moment of creation, stamped on the signed event payload, never recomputed downstream.

The lifecycle and the handle together let the engine classify every arriving event in O(1) and evict in batches when the window advances. The conceptual definitions live in [`concepts/event-lifecycle.md`](../../concepts/event-lifecycle.md) and [`concepts/birth-round.md`](../../concepts/birth-round.md); this lesson is the architectural lens onto how the current code wires those definitions into the per-event pipeline.

## Concept

The two concept files are short; read them before continuing if you have not — [`concepts/event-lifecycle.md`](../../concepts/event-lifecycle.md) for the staircase, [`concepts/birth-round.md`](../../concepts/birth-round.md) for the handle. Three framings to anchor on:

- **Four regions, two thresholds, one buffer.** From the consensus engine's perspective an event sits in one of four regions, indexed by its birth round relative to the running `EventWindow`:
  - *Future* — birth round greater than `EventWindow.getPendingConsensusRound()`. The hashgraph cannot link the event yet because the round it claims is still in front of the round consensus is currently building. The future-event buffer holds it.
  - *Admitted* — birth round in the half-open range `[ancientThreshold, pendingConsensusRound]`. The event is in the DAG. It contributes to round assignment, fame voting, and round-received computation in [A.1-2](A.1-2-rounds-and-witnesses.md) through [A.1-5](A.1-5-judges.md).
  - *Ancient* — birth round less than `ancientThreshold`. Removed from the hashgraph; possibly retained at outside-DAG sites if it has not yet crossed the expired threshold.
  - *Expired* — birth round less than `expiredThreshold`. Discarded everywhere a node controls; if a peer ever needs it the peer is past gossip and into reconnect territory.
    Both thresholds are birth-round values carried as fields on the `EventWindow` record. `expiredThreshold` is further back than `ancientThreshold`, so an event crosses *ancient* first and *expired* later; the interval between them is the operational catch-up window.
- **Birth round is the discriminator at every site.** `EventWindow.isAncient(event)` is `event.getBirthRound() < ancientThreshold` — three lines of code that every site downstream of the wire trusts. The future-event buffer's classifier is the symmetric `event.getBirthRound() > pendingConsensusRound`. The expired-side eviction in `ConsensusRounds` and the gossip `Shadowgraph` is the same comparison against `expiredThreshold`. There is no parallel structure for "non-deterministic ancient" or "ancient-by-NGen"; the entire lifecycle is birth-round-driven in current code.
- **The threshold values are output of the consensus algorithm.** The `EventWindow` that gates the *next* event is built into the `ConsensusRound` emitted by the *previous* `roundDecided`. [A.1-5](A.1-5-judges.md) walked how `roundDecided` derives `minimumJudgeInfoList` for the rolling ancient window and packages it together with the round number, the next pending round, the new ancient threshold, and the new expired threshold. So the lifecycle horizons advance in lockstep with consensus progress; they are not a parallel timer but the algorithm's own self-bookkeeping projected back onto the input stream.

A note on the paper. The paper describes ancient-ness against non-deterministic generation (NGen). The code's ancient horizon is birth round. The earlier code base named the thresholds `minGenNonAncient` and `minGenNonExpired` and computed them against per-event generations; current code retains the structural shape and replaces the discriminator. NGen still exists on `EventImpl` and is computed locally for use inside the algorithm (see [A.1-2](A.1-2-rounds-and-witnesses.md)) and inside the tipset selector (cluster A.4), but it does not gate the lifecycle.

## How it works

The lifecycle is enforced at three sites tied together by `DefaultConsensusEngine.addEvent` at [`DefaultConsensusEngine.java#L104`](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L104). The engine's three lifecycle-relevant fields — `linker`, `futureEventBuffer`, `consensus` — are constructed in the engine's constructor at [line 80–84](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L80); the future buffer is built with `FutureEventBufferingOption.PENDING_CONSENSUS_ROUND` ([line 84](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L84)), which fixes the future cutoff at `eventWindow.getPendingConsensusRound()` for this engine.

**Future cutoff.** The first thing `addEvent` does after the freeze short-circuit is hand the event to `futureEventBuffer.addEvent(event)` at [line 112](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L112). Inside the buffer ([`FutureEventBuffer.java#L83`](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-utility/src/main/java/org/hiero/consensus/event/FutureEventBuffer.java#L83)) three branches:
1. `eventWindow.isAncient(event)` returns `null` and the event is dropped silently — the buffer does not double-report ancient events that already cleared the upstream intake stages.
2. `event.getBirthRound() <= bufferingOption.getMaximumReleasableRound(eventWindow)` (i.e. `<= pendingConsensusRound` for the engine's option) returns the event unchanged — it is *admitted* relative to the future cutoff and falls through to the linker.
3. Otherwise the event is in a near-future round; the buffer files it in a `SequenceMap<Long, List<PlatformEvent>>` keyed by birth round and returns `null`. `addEvent` returns an empty `ConsensusEngineOutput` — the engine has accepted the event for later processing without producing any pre-consensus output yet.

**Ancient drop.** Future-cleared events enter the work loop at [line 126](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L126). Each iteration calls `linker.linkEvent(eventToAdd)` at [line 128](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L128). Inside the linker ([`ConsensusLinker.java#L79`](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/linking/ConsensusLinker.java#L79)) the first guard is `eventWindow.isAncient(event)` ([line 80](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/linking/ConsensusLinker.java#L80)); ancient events return `null`, the engine's `continue` at [line 131](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L131) skips them, and they leave no trace in the DAG. An admitted event is wrapped into an `EventImpl`, recorded in the linker's two parallel maps (the `birthRound`-keyed `SequenceMap` and the `Hash`-keyed `Map` from [A.1-1](A.1-1-hashgraph-dag.md)), and returned.

**Algorithm processing.** Admitted events flow into `consensus.addEvent(linkedEvent)` at [line 137](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L137); that is the entry point for the round / witness / fame / judge pipeline of [A.1-2](A.1-2-rounds-and-witnesses.md) through [A.1-5](A.1-5-judges.md), and may return zero or more `ConsensusRound`s.

**Window advance — the eviction half.** When `consensus.addEvent` returns a non-empty list of rounds, [line 179](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L179) takes the *last* round's `EventWindow` — the most advanced one — and calls `linker.setEventWindow(eventWindow)` at [line 183](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L183). Inside the linker ([`ConsensusLinker.java#L105`](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/linking/ConsensusLinker.java#L105)), `parentDescriptorMap.shiftWindow(eventWindow.ancientThreshold(), …)` ([line 109](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/linking/ConsensusLinker.java#L109)) walks the descriptors that just fell below the new ancient threshold; for each one the linker removes the entry from `parentHashMap` ([line 110](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/linking/ConsensusLinker.java#L110)), calls `event.clear()` to release the algorithm-state slots ([line 111](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/linking/ConsensusLinker.java#L111)), and accumulates the evicted `EventImpl` into the returned list. The engine then partitions that list at [lines 184–187](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L184) into events that reached consensus (no further action — they have already left on `consensusRoundOutputWire`) and events that did not — those are the *stale* events covered in [A.1-7](A.1-7-stale-events.md).

**Window advance — the release half.** Immediately after, [line 188](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L188) calls `futureEventBuffer.updateEventWindow(eventWindow)`. The buffer ([`FutureEventBuffer.java#L108`](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-utility/src/main/java/org/hiero/consensus/event/FutureEventBuffer.java#L108)) shifts its `SequenceMap`'s window forward to `bufferingOption.getOldestRoundToBuffer(eventWindow)` (i.e. `pendingConsensusRound + 1`); for every round entry that just fell below that floor, it walks the buffered events and, for each one not already ancient under the new window, appends it to a return list. The engine appends those events to its `eventsToAdd` queue and the loop iterates — buffered events that have just become admissible are linked, fed into consensus, and may themselves trigger another round-decision cycle. This is why a single call to `addEvent` may emit several decided rounds and process more events than the one that arrived: the future buffer can cascade-release on every window advance.

**Outside-DAG retention.** The expired threshold travels with `EventWindow` but the hashgraph itself does not honour it — by the time an event is past `expiredThreshold` it is already past `ancientThreshold` and has long since been evicted from the linker. The threshold is consulted at retention sites *outside* the hashgraph: the consensus-rounds buffer at `ConsensusRounds.getExpiredThreshold` ([`ConsensusRounds.java#L243`](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusRounds.java#L243)) and the gossip shadowgraph's `oldestUnexpiredIndicator` field ([`Shadowgraph.java#L67`](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-gossip-impl/src/main/java/org/hiero/consensus/gossip/impl/gossip/shadowgraph/Shadowgraph.java#L67)), seeded from `eventWindow.expiredThreshold()` at [line 121](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-gossip-impl/src/main/java/org/hiero/consensus/gossip/impl/gossip/shadowgraph/Shadowgraph.java#L121). The gossip side is the operational user of the gap: a peer that needs an event whose birth round is in `[expiredThreshold, ancientThreshold)` can still get it from this node's shadowgraph; a peer that needs an event with birth round below `expiredThreshold` cannot, and that fact is the local node's signal that the peer has fallen behind far enough to need a reconnect rather than gossip-driven catch-up. The full mechanics of that decision live in cluster B and cluster C; the lifecycle hand-off is just the threshold and the gap.

The handle itself — birth round — is on `PlatformEvent` at [`PlatformEvent.getBirthRound()`](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-model/src/main/java/org/hiero/consensus/model/event/PlatformEvent.java#L266); the sentinel for an unset value is [`EventConstants.BIRTH_ROUND_UNDEFINED`](https://github.com/hashgraph/hiero-consensus-node/blob/main/platform-sdk/consensus-model/src/main/java/org/hiero/consensus/model/event/EventConstants.java#L23). The field is part of the signed, immutable, gossiped section of an event (see the glossary entry at [`hashgraphGlossary.md#fields-of-an-event`](../../../../hashgraphGlossary.md#fields-of-an-event)); the creator stamps it once at event creation time as the value of the creator's pending round at that instant.

## Worked example

Run a node that has just decided round 100. The `EventWindow` packed into that round's `ConsensusRound` is `(latestConsensusRound = 100, newEventBirthRound = 101, ancientThreshold = 90, expiredThreshold = 80)`. `pendingConsensusRound` is `101`. Now four events arrive in close succession on `eventInputWire`.

- **Event `e_99`, birth round 99.** Future buffer: not ancient, `99 <= 101`, returned unchanged. Linker: not ancient under threshold 90; linked, inserted into both maps as an `EventImpl`. Consensus: enters the algorithm and may participate in fame voting on undecided witnesses for rounds 101 and beyond. Lifecycle state: *admitted*.
- **Event `e_85`, birth round 85.** Future buffer: not ancient (`85 >= 90`? — no, `85 < 90`, so `eventWindow.isAncient(e_85)` is true), returns `null`. The engine's null check at line 113 sees a `null` and returns an empty output. Lifecycle state: *ancient* (and it never reached the linker because the future buffer's first guard already handled it). Note: in this engine's wiring, the future buffer's ancient-drop branch is redundant with the linker's, because the next stage would have dropped the same event with the same reason; the buffer's check is structurally early but operationally indistinguishable.
- **Event `e_105`, birth round 105.** Future buffer: not ancient, `105 > 101`, filed under birth round 105 in the buffer's `SequenceMap`. The buffer returns `null`; the engine returns an empty output. Lifecycle state: *future*. The event will sit in the buffer until the window advances enough that `105 <= pendingConsensusRound`.
- **Event `e_77`, birth round 77.** Same path as `e_85`: future buffer's ancient branch fires, `null` returned. Lifecycle state: *expired* under any node that has the same window. Note that *expired* is not enforced separately in the consensus engine — the engine sees only the ancient drop. The expired-side fact is what the gossip shadowgraph and consensus-rounds buffer use; if `e_77` has somehow been on disk and a peer asked for it, neither retention site would still have it.

Now suppose `e_99` is the trigger that flips `RoundElections.isDecided()` true on round 101, [A.1-5](A.1-5-judges.md)'s `roundDecided` runs, and a `ConsensusRound` for round 101 emerges. Its `EventWindow` is `(latestConsensusRound = 101, newEventBirthRound = 102, ancientThreshold = 91, expiredThreshold = 81)` (numbers vary with the rolling judge window; the structural point is that thresholds advance by one per decided round in steady state). The engine takes that window and:

1. Calls `linker.setEventWindow`. The linker's `SequenceMap` shifts to round 91. Every `EventImpl` whose birth round is in `[…, 91)` is unlinked: removed from the hash map, `clear()`-ed, and added to the returned list. Suppose the unlinked set includes a self-event of this node, `s_88`, that was admitted long ago but never reached consensus; the engine partitions on `e -> !e.isConsensus()`, finds `s_88` non-consensus, and appends it to `staleEvents`. Stale-event reporting is the subject of [A.1-7](A.1-7-stale-events.md).
2. Calls `futureEventBuffer.updateEventWindow`. The buffer shifts its sequence map to `pendingConsensusRound + 1 = 103`. Buffered events with birth round in `[…, 103)` are released — there are none in this scenario because the only future event was `e_105` and `105 >= 103`. `e_105` stays buffered.

If a fifth event `e_102` arrives next: future buffer admits it (`102 <= 102`); the linker links it. If yet another round decides, lifting `pendingConsensusRound` to `103`, then on the next `setEventWindow` cycle the buffer's release scan covers the range up to `103` and `e_105`-buffered events whose birth round equals `102` would fall through; `e_105` itself remains buffered until the window passes `104`. (Spelling this out is to make the cascade structure concrete: every window advance is a chance for buffered events to drop into the linker, where they may themselves push consensus forward and trigger another window advance.)

Two corollaries the example surfaces. First, an event's lifecycle state is not a property of the event in isolation — it is a property of the event's birth round relative to *this node's current* `EventWindow`. Two nodes whose consensus has diverged in time will classify the same event differently for as long as the divergence persists. Second, the staircase is one-way: in the per-event pipeline an event only ever moves forward through *future → admitted → ancient → expired*. The buffer can release an event from *future* to *admitted* on a window advance, but no code path moves an event back from *ancient* to *admitted* — once `clear()` runs, the algorithm-state slots are gone and the event would have to be re-ingested as fresh input to be re-admitted. Reconnect, in cluster C, is exactly the path that does that re-ingestion; it does not retroactively ungranted ancient-ness, it loads a fresh snapshot whose `EventWindow` reframes the same event as admitted under a new threshold.

The relevant historical change worth flagging anyway is the discriminator swap from generation to birth round — the older code base named the thresholds `minGenNonAncient` and `minGenNonExpired` and computed them from event generations, and the [`concepts/event-lifecycle.md`](../../concepts/event-lifecycle.md) and [`concepts/birth-round.md`](../../concepts/birth-round.md) sections preserve that note.
