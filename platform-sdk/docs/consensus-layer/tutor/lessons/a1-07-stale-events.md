# Stale events

## Prerequisites

> - **`a1-06-birth-round-and-ancient`** — you know the ancient threshold sweeps upward as rounds decide, and that when it advances `ConsensusLinker.setEventWindow` evicts the newly-ancient events (calling `clear()` on each for GC) and returns them as a list, which `DefaultConsensusEngine.addEvent` then **splits** — releasing the events that reached consensus and routing the rest to the stale-events output. This lesson is that second half.
> - **`a1-05-judges-and-consensus-order`** — you know a decided round's judges fix a **total consensus order** that every honest node computes identically from the same graph, and that "reaching consensus" means being placed in that order. Stale is defined against this: a stale event is one that never made it in.

Beyond that, this lesson assumes `a1-03`'s **strongly-seeing** as a black box — "an event reaches consensus only once enough later events strongly see the witnesses below it, so an event no later event can reach can never get there" — and `a1-02`'s three round quantities. The new ground here is the half of a1-06's split that lesson deferred: what *stale* means, where the report goes, and why — unlike consensus order — it is a per-node observation rather than a network-wide fact.

## Incoming retrieval probes

This is the seventh mechanism lesson in Cluster A.1. Three things from earlier lessons resurface; the tutor should consolidate against them when they come up — these are authorial watch-for signals, not an opening quiz.

- **The eviction split**, from `a1-06-birth-round-and-ancient`. Canonical statement the tutor consolidates against: *when the ancient threshold advances, `setEventWindow` returns the newly-ancient events and the engine partitions them into those that reached consensus (released) and those that did not (sent to the stale output).* This lesson is the back half of that split; a firm grip on "the split happens at window advance" keeps the partition legible.
- **Consensus order as a deterministic, network-wide fact**, from `a1-05-judges-and-consensus-order`. Canonical statement: *a decided round's judges fix a total order that every honest node computes identically from the same graph; reaching consensus is a global fact, not a per-node one.* The whole threshold concept rests on contrasting the stale set against this — consolidate explicitly if the learner treats stale as just another agreed-upon output.
- **Strongly-seeing as the consensus gate**, from `a1-03-strongly-seeing` (held as a black box). Canonical statement: *an event reaches consensus only once enough later events strongly see the witnesses below it; an event that no later event can ancestrally reach can never get there.* Watch for it when the lesson explains why an event fails to reach consensus and ages out stale.

## Misconception watchlist

- **"Stale is a lifecycle state, like ancient or expired."** *(Over-generalization — state vs fate.)* Sounds like: "an event goes admitted → ancient → stale," using *stale* as the next rung after *expired*. Correction, in line: ancient and expired are **lifecycle states** an admitted event passes through ([event-lifecycle.md](../../concepts/event-lifecycle.md)); **stale is a fate** — the specific outcome of an admitted event whose lifecycle ended in *ancient* without ever reaching consensus. Most events that go ancient reached consensus first and are not stale ([stale-events.md](../../concepts/stale-events.md), [glossary: Stale](../../glossary.md#stale)).
- **"The stale set is the same on every node."** *(Adjacent import — assuming every hashgraph output is deterministic and agreed.)* Sounds like: "if `e` is stale it's stale network-wide," "stale is part of consensus." Correction: stale reports are **local observations** and are not deterministic across nodes — the partition reads *this node's* own admitted events and *this node's* own consensus view, both shaped by local gossip delivery ([glossary: Stale](../../glossary.md#stale), [stale-events.md](../../concepts/stale-events.md)). This is the lesson's threshold concept; the whole of Chunk 4 turns on it.
- **"An event dropped at intake for being too old is stale."** *(Over-generalization — conflating pre-admission rejection with a post-admission fate.)* Sounds like: "the events intake throws out for being ancient are the stale ones." Correction: stale is strictly **post-admission** — the event must have been admitted to this node's hashgraph and then aged out. An event that reaches intake already past the ancient threshold is **door-dropped** there (by the deduplicator, signature validator, or orphan buffer) and never enters the hashgraph, so it never becomes stale ([event-intake.md birth-round filtering](../../architecture/topics/event-intake.md#birth-round-filtering), [stale-events.md](../../concepts/stale-events.md)).
- **"There's a `StaleEventDetector` that scans for stale events."** *(Over-generalization from a legacy name.)* Sounds like: "where's the stale-event detector?" Correction: there is **no** `StaleEventDetector` class; the legacy `StaleEventDetectorOutput` enum still ships but is dead code (removal tracked in [hiero-consensus-node#25505](https://github.com/hiero-ledger/hiero-consensus-node/issues/25505)). Stale detection is the one-line partition `!e.isConsensus()` inside `DefaultConsensusEngine.addEvent`, computed synchronously when the window advances — not a separate scanner ([`StaleEventDetectorOutput` #L7-L16](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-model/src/main/java/org/hiero/consensus/model/event/StaleEventDetectorOutput.java#L7-L16), [`DefaultConsensusEngine.addEvent` #L188-L192](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L188-L192)).

## Mechanism

a1-06 drew the boundary of the DAG and showed the ancient threshold sweeping upward, evicting events and splitting the evicted ones into those that reached consensus and those that did not — then stopped at the split. This lesson takes up the second half: the not-yet-consensus evictions, which **are** the stale events ([hashgraph.md birth-round filtering](../../architecture/topics/hashgraph.md#birth-round-filtering), [stale-events.md](../../concepts/stale-events.md)). The spine has four chunks: what stale is (a fate, not a state), how stale events are produced (one predicate), where the reports go and who acts, and the property that makes stale unlike every other A.1 output — it is a local observation, not a consensus fact.

**Pre-training — terms this lesson integrates** (definitions live in the glossary and concept files; named here to set vocabulary):

- **Stale** — the fate of an admitted event that aged past the ancient threshold without ever reaching consensus ([glossary: Stale](../../glossary.md#stale), [stale-events.md](../../concepts/stale-events.md)).
- **`isConsensus`** — a per-event boolean on `EventImpl`, "is this part of the consensus order yet?", flipped to true by the algorithm when the event is placed in a round's consensus order ([`EventImpl.isConsensus` #L167-L172](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/EventImpl.java#L167-L172)).
- **Consensus order** — the total order a decided round fixes; reaching it is exactly what a stale event never did (recall `a1-05`) ([glossary: Consensus order](../../glossary.md#consensus-order)).
- **Self-event** — an event this node authored; the actionable stale case, because the node owns its transactions and can resubmit them ([glossary: Self-event](../../glossary.md#self-event)).
- **Stale-events output wire** — the `OutputWire<PlatformEvent>` carrying stale events out of the engine to a single execution-layer consumer ([`HashgraphModule.staleEventOutputWire` #L83-L89](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph/src/main/java/org/hiero/consensus/hashgraph/HashgraphModule.java#L83-L89)).
- **Prehandle** — the execution layer's early per-event pass over transactions, before the round reaches consensus; named here because a prehandled event that goes stale must be reported back (a forward pointer, paid off in Chunk 3) ([glossary: Prehandle](../../glossary.md#prehandle)).
- **Quiescence** — an opt-in pause feature (Cluster A.4) that constrains how stale events are reported across the Consensus/Execution boundary; named here only as a forward pointer ([glossary: Quiescence](../../glossary.md#quiescence)).

### Chunk 1 — Stale is a fate, not a state `{moment: m1-fate}`

a1-06 left the admitted → ancient → expired **lifecycle**: ancient and expired are *states* an admitted event passes through as the window sweeps past it ([event-lifecycle.md](../../concepts/event-lifecycle.md)). Stale is not on that ladder. Stale is the **fate** of an event whose lifecycle ended in *ancient* without ever reaching consensus — its transactions never gained consensus order and, by definition, never will ([stale-events.md](../../concepts/stale-events.md), [glossary: Stale](../../glossary.md#stale)). Most events that go ancient reached consensus *first* and are not stale; stale is the specific minority that aged out unordered.

Two boundaries fix the definition, and both matter. First, stale is **post-admission**: the event was admitted to this node's hashgraph — linked into the DAG, the middle gate of a1-06 — and *then* aged out. An event that arrived at intake already past the ancient threshold is door-dropped there (by the deduplicator, signature validator, or orphan buffer) and never enters the hashgraph, so it is not stale; it was simply rejected ([event-intake.md birth-round filtering](../../architecture/topics/event-intake.md#birth-round-filtering)). Second, stale is **terminal**: a stale event will never reach consensus, because it is now ancient and no later event can reach it to give it the strongly-seeing it would need (`a1-03`).

**Load-bearing line (signal):** stale is *admitted, then aged out without consensus*. The two ways to be ancient and **not** stale are (a) reached consensus before going ancient, and (b) never admitted in the first place. Keep both exclusions in view — every later section rests on "admitted, then aged out unordered."

### Chunk 2 — How stale events are born: one predicate `{moment: m2-split}`

a1-06 showed that when a round decides, the engine calls `linker.setEventWindow`, which evicts the newly-ancient events, calls `clear()` on each for GC, and **returns them as a list** ([`ConsensusLinker.setEventWindow` #L105-L116](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/linking/ConsensusLinker.java#L105-L116)). The stale events are exactly the half of that returned list that never reached consensus. `DefaultConsensusEngine.addEvent` partitions it with a single predicate — `ancientEvents.stream().filter(e -> !e.isConsensus()).map(EventImpl::getBaseEvent).forEach(staleEvents::add)` ([#L188-L192](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L188-L192)).

The predicate is `!e.isConsensus()`. `isConsensus()` is a per-event boolean on `EventImpl` — "is this part of the consensus order yet?" — flipped to true by the algorithm the moment the event is placed in a round's consensus order ([`EventImpl.isConsensus` #L167-L172](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/EventImpl.java#L167-L172), [`ConsensusImpl` sets it true #L836](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/consensus/ConsensusImpl.java#L836)). So a just-evicted event carries a reliable answer to "did this node ever order it": the events for which the predicate holds (`isConsensus == false`) are the stale ones; the events that fail it (already consensus) are simply released, nothing further.

**Structural detail (grounded in code, not paraphrase):** this is **synchronous**, not periodic. There is no background task scanning the DAG for stale events. The partition runs inline inside `addEvent`, in the same call where a decided round advanced the window: `setEventWindow` returns the newly-ancient events and the very next statements append the non-consensus ones to a local `staleEvents` list ([#L129, L188-L192](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L188-L192)), which is returned as one of the three fields of `ConsensusEngineOutput`. The wired module turns that field into the stale output wire with a plain transformer-then-splitter — no timer, no scheduler ([`DefaultHashgraphModule` #L77-L80](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultHashgraphModule.java#L77-L80)). Note too that `clear()` severs an evicted event's parent links but does **not** touch its `isConsensus` flag, so the partition reads a still-valid boolean even on a just-cleared event ([`EventImpl.clear` #L398-L407](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/EventImpl.java#L398-L407)).

**Load-bearing line (signal):** one predicate, `!e.isConsensus()`, computed the instant the window advances. Stale is not found by a scanner; it is the **leftover of the eviction split**, decided synchronously inside `addEvent`.

### Chunk 3 — Where stale reports go, and who acts `{moment: m3-consume}`

The stale list leaves the engine on a single wire, `staleEventOutputWire()`, typed `OutputWire<PlatformEvent>` ([`HashgraphModule.staleEventOutputWire` #L83-L89](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph/src/main/java/org/hiero/consensus/hashgraph/HashgraphModule.java#L83-L89)). It has exactly **one consumer**: a callback in the execution layer, registered through `PlatformBuilder.withStaleEventCallback(Consumer<PlatformEvent>)` ([#L240-L255](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-platform-core/src/main/java/com/swirlds/platform/builder/PlatformBuilder.java#L240-L255), soldered at [`PlatformWiring` #L134-L138](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-platform-core/src/main/java/com/swirlds/platform/wiring/PlatformWiring.java#L134-L138)). The consensus layer delivers the event and does nothing else — it resubmits no transactions itself.

What the execution layer does depends on who created the event. For a **self-event** — one this node authored — the transactions never made it into the consensus order, so the execution layer may resubmit them on a fresh self-event; the callback's own JavaDoc says exactly this ("it may be a good idea to resubmit the transactions in the stale event") ([#L240-L255](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-platform-core/src/main/java/com/swirlds/platform/builder/PlatformBuilder.java#L240-L255)). For a **peer-authored** stale event the report is informational: the local node does not own those transactions and has nothing to resubmit ([stale-events.md](../../concepts/stale-events.md)). One caveat the JavaDoc flags: stale detection catches every stale self-event only while the node stays online; an event that goes stale "in the gap" across a restart or reconnect may not be detected.

There is one more reporting requirement, and it is easy to mis-model. An event that the execution layer already saw on its **prehandle** path must have its stale outcome reported through to the execution layer, so prehandle state can be reconciled ([glossary: Prehandle](../../glossary.md#prehandle)). It is tempting to imagine a wire from the stale output into the prehandler that does this. **There is none** — the stale wire is soldered only to the execution layer's callback ([`PlatformWiring` #L134-L138](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-platform-core/src/main/java/com/swirlds/platform/wiring/PlatformWiring.java#L134-L138)). The guarantee is instead a **contract** the consensus engine owns: every event it emits on the pre-consensus wire — which is what the execution layer prehandles from — is guaranteed to later resolve to *either* a consensus round *or* the stale output, both of which the engine owns. So a prehandled event's fate always reaches the execution layer by one of those two terminal paths; no dedicated stale→prehandle wire is needed ([`PlatformWiring` #L143-L147](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/swirlds-platform-core/src/main/java/com/swirlds/platform/wiring/PlatformWiring.java#L143-L147)). The precise reconciliation, and how it tightens under **quiescence**, belongs to Cluster A.4 ([glossary: Quiescence](../../glossary.md#quiescence)).

**Load-bearing line (signal):** one wire, one execution-layer consumer, **no stale→prehandle wire**. The prehandle guarantee is an engine contract (pre-consensus ⇒ consensus-or-stale), not a wired path.

### Chunk 4 — Stale is local; consensus is global `{moment: m4-nondeterminism}`

Every output Cluster A.1 has produced so far is a **consensus fact**: a decided round's judges, its consensus order, an event's round-received — every honest node computes them identically from the same graph, with no messages exchanged (`a1-05`). The stale set is the first output that is **not** like that. The partition `!e.isConsensus()` reads two things, and both are *local*: this node's own linker contents (which events it admitted) and this node's own consensus observation (which of them it ordered) ([`DefaultConsensusEngine.addEvent` #L188-L192](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L188-L192)). Neither is a network-wide fact.

So the same event can meet different fates on different nodes. A node reports `e` stale only if it admitted `e` and then aged it out without ordering it. A better-connected node may have had `e` reach consensus — `e` was propagated there well enough to land in the ancestry of a decided round's judges — and reports nothing. A node that never received `e` over gossip never admitted it, has nothing to evict, and never reports it. The input that differs is **what gossip delivered** to each node, which is not a consensus-level fact ([stale-events.md](../../concepts/stale-events.md), [glossary: Stale](../../glossary.md#stale)). The stale-events output is a local-observation stream.

What does **not** differ is the consensus order itself. An event's transactions are in the consensus order on every honest node or on none — that is global. Staleness never contradicts it: a local stale report says only that *this* node, having admitted `e`, did not observe it reach the global order before `e` went ancient here. *(See the Contrasting cases material below, which the tutor draws on at this moment.)*

**Load-bearing line (signal):** the partition reads **local linker + local consensus**, so the stale set is a per-node observation. Consensus order is global; the stale set is local; the bridge invariant is *ordered on all honest nodes or none*.

## Engagement moves

### Moment `m1-fate`

*Why load-bearing (exposition):* the entry risk is the state-vs-fate conflation a1-06 flagged and the pre-admission/post-admission boundary; both must be solid before the partition is built on top of them.

**Move A — free recall.** *Diagnosis tag:* the learner needs the fate-vs-state distinction and the admission boundary articulated in their own words.
- Prompt (verbatim): "Define a stale event in one sentence. Then name the two different ways an event can end up below the ancient threshold and *not* be stale — one for an event that was once in this node's hashgraph, one for an event that never was."
- `canonical_answer`: A stale event is one that was admitted to this node's hashgraph and then aged past the ancient threshold without ever reaching consensus (and never will). The two non-stale ways below the threshold: (1) the event **reached consensus** before it went ancient — most ancient events did; (2) the event arrived at intake **already ancient** and was door-dropped there, never admitted, so it never became stale ([stale-events.md](../../concepts/stale-events.md), [event-intake.md birth-round filtering](../../architecture/topics/event-intake.md#birth-round-filtering)).
- `alternative_correct_answers`:
  - "Stale = admitted then aged out without consensus. Not stale if it reached consensus first, or if it was never admitted (dropped at intake as already-ancient)."
  - "An admitted event that went ancient without being ordered. The two exclusions: already-ordered ancient events, and events rejected at intake for arriving too old."

**Move B — direct walk with cued check.** *Diagnosis tag:* the learner is clearly fluent on a1-06's lifecycle and needs only a verify before the partition.
- Prompt (verbatim): "Ancient and expired are lifecycle states. Is stale a third state on that same ladder — admitted → ancient → expired → stale — or is it something else? Say what it is in one phrase."
- `canonical_answer`: Not a state on the ladder. Stale is a **fate** — the outcome of an admitted event whose lifecycle ended in *ancient* without ever reaching consensus. Ancient and expired are states every admitted event can pass through; stale is the specific outcome of one that aged out unordered ([glossary: Stale](../../glossary.md#stale), [event-lifecycle.md](../../concepts/event-lifecycle.md)).
- `alternative_correct_answers`:
  - "Something else — a fate, not a state; the result of going ancient without consensus."
  - "Not a fourth state; it's the fate of an admitted event that went ancient unordered."

### Moment `m2-split`

*Why load-bearing (exposition):* the partition predicate and its synchronous timing are the lesson's central new structural facts, and the "a scanner periodically finds stale events" model is the trap to head off.

**Move A — worked example with self-explanation.** *Diagnosis tag:* the learner is new to reading the partition (the common case here).
- Walk (exposition): a round decides and raises the ancient threshold; `setEventWindow` returns the just-evicted events as a list; `addEvent` filters that list with `e -> !e.isConsensus()` and sends the survivors to the stale output. `isConsensus` is a per-event boolean set true when the algorithm placed the event in consensus order. **Load-bearing line:** the stale events are the evicted events whose `isConsensus` is *false*.
- Self-explanation prompt (verbatim): "The engine has the list of events that just became ancient, and it keeps only the ones for which `!e.isConsensus()` holds. What does that boolean being *false* tell you about the event — and why is reading it *at eviction time* enough to be sure the event is stale rather than just late?"
- `canonical_answer`: `isConsensus == false` means the algorithm never placed the event in any round's consensus order. Reading it at eviction is conclusive because the event is being evicted precisely for going ancient — its birth round fell below the ancient threshold — and an ancient event can never again be ancestrally reached by new events to gain the strongly-seeing it would need to reach consensus (`a1-03`). So "not consensus yet, **and now ancient**" means "not consensus ever" ([`EventImpl.isConsensus` #L167-L172](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/EventImpl.java#L167-L172), [`DefaultConsensusEngine.addEvent` #L188-L192](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L188-L192)).
- `alternative_correct_answers`:
  - "False means it never got a place in consensus order; at eviction it's going ancient, so later events can't reach it anymore and never will — stale, not merely late."
  - "It was never ordered, and because it's now ancient it's past the point where later events could order it, so the fate is final."
- `followup` (if the learner says only "it didn't reach consensus" without the terminal/why-now part): "Right that the flag means not-yet-ordered. Now say why *now* makes it final: what does an event need from later events to reach consensus, and can an event that just went ancient still get it?"

## Contrasting cases material

The threshold concept is *the stale set is a per-node local observation, unlike consensus order, which is a deterministic network-wide fact.* The two cases hold "an output the machinery produces about event `e`" constant and vary the input it reads.

- **Case 1 — consensus order (deterministic, global).** When a round decides, its judges fix a total order over the events below them, and every honest node computes the identical order from the same graph, with no messages exchanged (`a1-05`). The input is *shared graph structure*; the output is derived and agrees network-wide ([judges.md](../../concepts/judges.md), [glossary: Consensus order](../../glossary.md#consensus-order)). *Surface:* computed from the replicated DAG. *Who agrees:* every honest node.
- **Case 2 — the stale set (non-deterministic, local).** The partition `!e.isConsensus()` runs against *this node's* own linker contents and its own consensus observation — what it admitted and what it ordered ([`DefaultConsensusEngine.addEvent` #L188-L192](https://github.com/hiero-ledger/hiero-consensus-node/blob/main/platform-sdk/consensus-hashgraph-impl/src/main/java/org/hiero/consensus/hashgraph/impl/DefaultConsensusEngine.java#L188-L192)). A node reports `e` stale only if it admitted `e` and aged it out unordered; a better-connected node may have ordered `e`, and a node that never received `e` is silent ([stale-events.md](../../concepts/stale-events.md), [glossary: Stale](../../glossary.md#stale)). *Surface:* computed from local observation. *Who agrees:* no cross-node guarantee.

**The deep invariant that survives the surface differences:** an event's transactions are in the consensus order on every honest node or on none — consensus is global. Both outputs ultimately concern that one global fact: consensus order *is* it, and the stale set is each node's local *observation* of whether a particular event reached it before going ancient locally. The non-determinism is purely in observation, not in the underlying consensus fact. *(Why a poorly-connected event fails to reach consensus — it never lands in the ancestry of a decided round's judges — is `a1-03`/`a1-05` material; this case set is only about why the *report* of that failure is local.)*
